# 画面遷移図の表示方式 — Mermaid UML 4タブ構成（最終決定）

## 文脈

GUIの画面遷移図をどう表示するかについて、以下の経緯を経て最終決定に至った。

**当初の問題**: Mermaid シーケンス図で28画面を描くと参加者が28列・メッセージ97本になり、実用不能なほど読めなかった。

**一時的なvis.js切替（廃止）**: vis.jsのインタラクティブネットワーク図へ切り替えを試みたが、JSTQB FL レベルのQAテスターが使う現場では「図と表を並べて番号で照合する」という使い方が主であり、ズーム・ドラッグ操作よりも読みやすさと照合性が優先されると判断した。

## 決定

**GUIの画面遷移図タブは Mermaid UML 4サブタブ構成で実装する（vis.jsは採用しない）。**

実装ファイル: `static/js/view-transition.js`（432行）

### 4サブタブ構成

| サブタブ | 内容 |
|---------|------|
| シーケンス図 | 代表的な操作順遷移を参加者ごとにシーケンスで表示 |
| コミュニケーション図 | 画面間の全体関係を番号付きエッジで表示 |
| アクティビティ図 | テスト手順（操作→期待結果）をアクティビティとして表示 |
| テスト観点マップ | 遷移を4QA観点（到達性・入力後遷移・認証導線・業務クリティカル）に分類 |

### 共通ルール

- 全クロール済みページの50%以上から出るリンクは共通ナビゲーション（ヘッダー/フッター）と判定して非表示
- フォーム送信は「フォーム送信」イベントとしてリンク遷移と区別
- 各図のエッジ番号（T01, T02...）と対応表の No を揃える
- ズームコントロール（−/100%/＋/Fit）を全サブタブに設置

## 棄却した代替案

- **vis.js ネットワーク図**: インタラクティブで視覚的に優れるが、QAテスターが「図を見ながらテストケースを書く」用途では照合性が低い。エッジ番号と表のNoを紐付ける設計が困難。
- **D3.js**: 実装コストが高い。

## 帰結

- `src/generator/mermaid_generator.py` と `transition.mmd` はCLIエクスポート用として残す（後方互換）
- GUIの遷移図は `static/js/view-transition.js` で Mermaid CDN（jsdelivr）から遅延ロード
- vis.js 関連コードは削除済み
- オフライン環境では Mermaid CDN が届かないため図は表示されない（フォールバック: Mermaidソース表示）
