## Phase 1: 信頼の再建（最優先・全ペルソナに波及）

**狙い**: ツールが出す数字を、第三者検証会社が顧客にそのまま出せる水準にする。

### 1-A. 構造ベース画面同定（C-1の根治）
- **問題**: `reserve.html?plan-id=N` の同一フォームを別画面計上 → 28画面（実16）/222項目（実54）/808ケース/135時間が約4倍。
- **設計**: クエリ除去パス＋フォーム/フィールド構造ハッシュで画面をグルーピング。代表1件＋バリエーション数を保持。データ層（`report.json`）に正規化情報を付与し、**GUI/HTML/PDF/Excel 全消費者が重複統合後の数値で計算**。
- **データ契約（report.json 拡張）**:
  - 各 screen に `canonical_key` / `is_canonical` / `variation_count` / `variation_urls`
  - `meta.screen_count`（統合後）を追加（`page_count`=生のまま）
- **UI**: KPIを「28ページ検出 → 16画面（クエリ重複を統合）」と二段表示。リスク上位・工数概算は統合後で再計算。
- **担当**: バックエンド（新モジュール＋json_reporter＋テスト）= **Codex**／JS・HTML・目視検証 = **Claude**

### 1-B. 工数概算の透明化（m-3）
- `1ケース10分`固定をやめ、係数を `meta` に出し、サマリーに前提を脚注表示。将来は設定で可変。

### 1-C. 「期待を超える」加点: 信頼性バッジ
- 概要に「**この数値は重複統合済み・機械導出（監査可能）**」バッジ＋内訳を表示。検証会社が顧客に出す時の安心材料。

**完了条件**: `make test` + `verify.sh` GREEN ／ 概要KPIが16画面・統合後ケース数 ／ 目視で「28→16統合」表示確認。

## Phase 3: AIの実体化（差別化・高難度）

**狙い**: 「今後実装」を消し、P3（事業化）とP4（技術）双方を満たす。

### 3-A. LLMテスト観点生成を1ステップ実配線（M-1）
`src/llm/` の既存モジュール＋`openai_qa.py` を QAプロセスの「テスト分析」1ステップに実接続。OFF時は決定的ローカル生成（無料）を維持。

### 3-B. 「決定的 vs AI補完」の可視化
出力の各項目に出自バッジ（⚙️決定的導出／✨AI補完）。P4の「過剰宣伝するな」に応え、監査可能性を担保。

### 3-C. プロバイダ抽象（m-4）
OpenAI固定をやめ、`LLMProvider` インターフェース（OpenAI/Anthropic/ローカル差し替え可）。ベンダーロック回避。

**担当**: 設計・プロンプト・抽象 = **Claude**／実装スライス = **Codex**。
