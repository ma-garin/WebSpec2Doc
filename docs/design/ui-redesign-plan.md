# 設計ドキュメント: 画面デザイン刷新計画（galleryinline.html ベース）

対象: WebSpec2Doc フロントエンド（`templates/` / `static/`）
作成日: 2026-07-15
状態: **レビュー待ち（未実装）**
デザイン源: `~/Downloads/galleryinline.html`（Harness ToDo モックのデザインシステム）

---

## 1. 背景と結論

`galleryinline.html` を基準に UI を刷新できるか検討した。結論:

- **技術的に可能**。かつ**ゼロからの作り直しではなく既存資産の磨き上げ**になる。
- 決め手: 現行 WebSpec2Doc（`static/tokens.css`）とモックが**同一のデザイン DNA** を共有している。
  - プライマリ `#1976D2` / `#0D47A1`（完全一致）、Noto Sans JP + JetBrains Mono、優先度色・スペーシング・角丸のトークン思想が同じ。
  - シェル構造（サイドバー＋トップバーの SPA）も一致。現行 `nav.html` / `topbar.html` は既に BEM 風（`app-nav-item` 等）。

## 2. 原則（最重要）

1. **移植するのは「デザイン言語とコンポーネント」であって「画面」ではない。**
   モックは別プロダクト（ToDo 管理）。ToDo 一覧・カレンダー・プロジェクト等を丸写ししない。
   WebSpec2Doc のドメイン（crawl → generate → QA/viewpoints → traceability）に当てはめ直す。
2. **既存トークン/コンポーネントを拡張**する（新設計システムを並置しない）。`tokens.css` を単一の真実源に保つ。
3. **各フェーズ完了時点で E2E をグリーンに戻す**（ビジュアル回帰スナップショット再ベースライン込み）。常時リリース可能な状態を維持。
4. **イミュータブル/多数小ファイル原則**（`common/coding-style.md`）を CSS 分割にも適用。
5. 直近で追加した認証画面（`templates/auth/` + `static/css/auth.css`）は既にモック準拠。**これを刷新の参照実装（リファレンス）とする**。

## 3. ギャップ分析（現行 → モック目標）

| 領域 | 現行 | モック目標 | 差分の性質 |
|---|---|---|---|
| トークン | 定義済み・整合 | 同じ | ほぼ差分なし（微調整） |
| シェル（sidebar/topbar） | 実装済み・グループ分けあり | 検索バー・ユーザーアバター・折りたたみが洗練 | 磨き上げ |
| KPI/統計カード列 | dashboard hero 中心 | 4 枚の stat カード列（アイコン+数値+補足） | 新コンポーネント化 |
| データテーブル | 一覧は簡易 | 列フィルタ・タブ・ページネーション・状態バッジ | 昇格（要注力） |
| バッジ/チップ | 優先度色トークンあり | 優先度チップ・状態バッジ・ロールバッジ | 部品統一 |
| アバター | 一部 | イニシャル配色アバター | 部品化 |
| アクティビティ/フィード | なし | タイムライン風フィード | 新規（run-history に流用可） |
| 空/エラー/モーダル状態 | 一部あり | 網羅（空 0 件・エラー・確認ダイアログ・通知） | 状態の体系化 |
| ダークモード | トグルあり | ライト/ダーク両対応の設計 | 全コンポーネントで担保 |

## 4. コンポーネント・マッピング（モック → WebSpec2Doc 適用先）

| モックの部品 | WebSpec2Doc での適用先 |
|---|---|
| stat カード列 | dashboard（解析数/成功率/実行中/要対応）・run-history サマリ |
| フィルタ付きデータテーブル | run-history、testcases、traceability、viewpoints 一覧 |
| 状態バッジ（進行中/完了/超過） | AutoRun 実行状態、crawl 進捗、testcase 状態 |
| 優先度チップ（高/中/低） | 品質観点・testcase の重大度（既存の critical/high/medium/low トークンと接続） |
| アクティビティフィード | run-history のタイムライン表示 |
| 検索バー（トップバー） | サイト/成果物横断検索（将来。まずは UI 骨格のみ） |
| 空/エラー状態 | 全一覧ビューの 0 件・失敗表示 |
| 確認ダイアログ/通知 | 削除確認、生成完了通知（既存の完了通知を刷新） |

## 5. フェーズ計画（Sonnet 実行可能な粒度）

各フェーズは独立ブランチ → PR → E2E 再ベースライン → main マージを 1 単位とする。

### Phase 0: 基盤（トークン/コンポーネント CSS の整備）
- `static/tokens.css` をモックと突き合わせ、不足トークン（stat カード/バッジ/アバター用）を追加。
- 共通コンポーネント CSS を `static/css/` に分割整理（例: `components-cards.css` / `components-table.css` / `components-badges.css`）。
- 変更ファイル: `static/tokens.css`, `static/css/*`（新規分割）。
- E2E 影響: 小（視覚微差）。ビジュアル基準線を再取得。

### Phase 1: アプリシェル（sidebar / topbar）
- `nav.html` / `topbar.html` をモックのシェルに合わせて洗練（検索バー骨格・アバター・折りたたみ挙動）。
- 変更: `templates/partials/nav.html`, `templates/partials/topbar.html`, 対応 CSS/JS（`static/app.js` の該当箇所）。
- E2E 影響: 中（多くのビジュアル基準線が変わる）。

### Phase 2: ダッシュボード
- hero を保ちつつ stat カード列＋最近のアクティビティ（run-history 由来）を追加。
- 変更: `templates/partials/view-dashboard.html`, 関連 CSS/JS。

### Phase 3: データ重めビュー（run-history → testcases → traceability → viewpoints）
- フィルタ付きデータテーブル・状態バッジ・ページネーションへ順次昇格。1 ビュー = 1 コミット。
- 変更: 各 `view-*.html` と対応 JS。

### Phase 4: 状態とモーダル
- 空/エラー/確認ダイアログ/完了通知を体系化して差し替え。

### Phase 5: ダーク＆最終ポリッシュ
- 全コンポーネントのダーク対応・アクセシビリティ（コントラスト/フォーカスリング）最終確認。

## 6. 検証ゲート戦略（最大のコスト源）

UI を触ると以下が発火するため、フェーズごとに必ず回す:

1. `make test`（ユニット/統合）緑を維持。
2. `make verify-ui`（E2E）。**ビジュアル回帰はフェーズ意図に沿って再ベースライン**:
   `pytest tests/e2e/ --update-snapshots` → 差分を目視確認 → コミット。
3. E2E セレクタが現行 DOM に依存する箇所は、DOM 変更に追従修正。
4. pre-commit UI ハッシュゲートは `make verify-ui` 成功で `.ui-verified` マーカーが更新される。

> 注: 現在 main は認証と無関係の既存 E2E 5 件が赤（autorun modal / history diff / realbrowser crawl / visual regression）。刷新着手前にこの 5 件を「別チケットで解消」または「既知 flaky として除外運用」を決める必要がある（第8章 Q3）。

## 7. スコープ外
- モックの ToDo 固有画面（ToDo 一覧・カレンダー・プロジェクト・マイページ）の丸写し。
- 検索・通知の実バックエンド機能（本計画は UI 骨格のみ。機能は別途）。
- 認証画面の再デザイン（既にモック準拠。トークン整合の微調整のみ）。

## 8. レビューで決めたいこと
- **Q1**: フェーズ順序と粒度（Phase 0→5）でよいか。優先したいビューはあるか（例: run-history を先に）。
- **Q2**: ビジュアル回帰は「フェーズごとに再ベースライン（推奨）」でよいか。
- **Q3**: 着手前提として、既存赤 E2E 5 件を (a) 別チケットで解消 / (b) 既知 flaky として除外、どちらにするか。
- **Q4**: 実行体制（このまま段階実装を Claude が進める / 各フェーズでレビューを挟む）。

## 9. リスク
- ビジュアル基準線の大量再取得に伴う「意図しない差分の見落とし」。→ フェーズを小さく保ち目視確認を必須化。
- E2E セレクタ追従漏れによる回帰。→ フェーズごとに `make verify-ui` を必須ゲート化。
- デザイン言語の“取り違え”（別プロダクトの画面を踏襲してしまう）。→ 第2章原則1を各 PR のレビュー観点に固定。
