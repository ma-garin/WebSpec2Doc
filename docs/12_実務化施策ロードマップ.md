## 採用基準

すべての施策は次の2点を満たすこと:

1. **顧客の QCD のどこに効くか**を特定できる（Q=品質 / C=コスト / D=納期・速度）
2. **主張できること／できないことの境界**を明示する（evidence-only 原則の拡張。ツールが観測した事実のみを主張し、十分性・網羅性は主張しない）

### 不採用とした案（理由の記録）

| 案 | 不採用理由 |
|---|---|
| リスクベース回帰テスト（影響テストのみ実行） | 「これだけやればいい」という十分性の主張になり、クロールで観測できない変更（バックエンドロジック・データ起因）の責任をツールが負えない。下界提示（実行順序の並べ替え）に縮退させる場合のみ将来再検討 |
| 新人QA演習モード | 顧客のテストプロジェクトの QCD に効かない（発注側の教育予算の話になる）。購買理由にならない |
| 本番/ステージング環境差分 | ユーザー判断で不要 |
| バグ報告書ジェネレータ | ユーザー判断で不要 |

## P0: 実務投入に必須（信頼の底）

監視ツールは「実行されなかったことに気づける」ことが信頼の底線。現状はクロール失敗がサイレントに沈む。

### P0-1. スケジューラの運用成熟化 — Q/D

- **内容**: (a) 実行時間帯・曜日ウィンドウ（「平日 2:00-5:00 のみ」等）、(b) 失敗時リトライ（回数上限＋バックオフ）、(c) クロール失敗の通知、(d) 実行履歴の永続化と UI 表示
- **根拠**: changedetection.io（OSS 無料）ですらタイムゾーン＋曜日/時間帯制限を標準装備
- **実装ポイント**: `web/services/scheduler.py` の `_calc_next_run_at` をウィンドウ対応に拡張。`_run_crawl` の失敗パスで `web/services/notifier.py` を呼ぶ。cron 式フル対応は不要（daily/weekly/monthly＋時間帯ウィンドウで足りる）

### P0-2. 通知の実務化 — D

- **内容**: (a) Microsoft Teams 対応、(b) 通知文面の Jinja2 テンプレート化、(c) diff サマリ（追加/削除ページ名の上位 N 件）の本文埋め込み、(d) テスト送信ボタン、(e) クロール失敗通知（P0-1(c) と共通）
- **根拠**: changedetection.io は apprise + Jinja2 が標準。Fluxguard は「メールを開かなくても分かる」diff 埋め込みを売りにする
- **実装ポイント**: `web/services/notifier.py` の `_build_text` をテンプレート化、`NotifierConfig` に `template` を追加

### P0-3. 初回オンボーディングツアー — C

- **内容**: 初回ログイン時に「STEP1: ここをクリック」型のガイド付きツアー。(a) ウィザード4ステップを3〜5個のツールチップで巡回、(b) 「最初のサイト登録→初回クロール→レポート閲覧」チェックリスト、(c) サイト未登録時の empty state に同梱デモサイトへの導線
- **ライブラリ**: **driver.js 一択**（MIT・5KB gzip・依存ゼロ・ローカル同梱可）。Shepherd.js / Intro.js は AGPL＋商用有償のため不採用
- **設計指針**（Appcues/Pendo 一次資料）: ツールチップは1つ140字以内、「操作方法」でなく「価値」を語る。Orient→Activate→Reinforce の3段階
- **実装ポイント**: `static/` に driver.js を同梱、利用者レコードに `tour_completed` フラグ、チェックリストの初回成功はデモサイト（`make demo`）クロールへ誘導

### P0-4. データ保持ポリシーとバックアップ手順 — 運用リスク解消

- **内容**: (a) スナップショット世代管理（保持世代数 or 期間の設定と自動削除。既定は無制限のまま）、(b) ディスク使用量表示、(c) バックアップ/リストア手順書（`output/` と `instance/` の対）
- **根拠**: Wachete 12ヶ月・Fluxguard 永久アーカイブと、保持期間は製品仕様として明示するのが通例。現状は無制限蓄積でディスク枯渇リスク
- **実装ポイント**: スケジューラループに世代 GC を追加。削除は監査ログに記録

### P0-5. 管理操作の監査ログ — 企業導入の前提

- **内容**: ログイン成否・メンバー/ロール変更・スケジュール/通知設定変更・エクスポートをテナント単位で記録し、admin が閲覧できるビューを提供
- **根拠**: EnterpriseReady「Audit Logs」。現状の `audit.jsonl` はクロール行為のみ
- **実装ポイント**: 既存と同形式（JSONL）で `instance/tenants/{slug}/admin_audit.jsonl` に追記

## Quick Win: ついで検査ファミリー（低工数・即効・Time to Value）

クローラーは全ページを訪問済みなので、追加クロールなしで成果物を増やせる。導入初日の説得力に直結する。

### QW-1. 技術ヘルスチェックの同梱（案14） — Q

- **内容**: 404/500・リンク切れ・JS コンソールエラー・混在コンテンツを収集し、レポートに1タブ追加
- **主張境界**: 到達したページで観測されたもののみ
- **実装ポイント**: `src/crawler/page_crawler.py` でレスポンスステータスと `page.on("console")` を収集、レポートタブ追加は `src/generator/html_reporter.py`

### QW-2. アクセシビリティ監査タブ（案9） — Q/C

- **内容**: クロールついでに画面ごとの WCAG 違反を axe で検出し、レポートに1タブ追加。日本の公共・自治体案件では JIS X 8341-3 対応が事実上必須
- **主張境界**: axe が機械判定できる項目のみ（WCAG 達成基準の約半分。残りは人の判断が必要と明記）
- **実装ポイント**: `src/ux/assets/axe.min.js` は同梱済み。クロール時に `page.evaluate` で実行し結果を evidence として保存

## P1: 商用競争力（差別化の中核と成果物拡充）

### P1-1. 文書駆動モード＋ルールベース MBT パイプライン（ユーザー案2） — Q/C・最大の差別化

- **内容**: 仕様書・要件文書を取り込み、テスト方針→テスト基本設計→テスト詳細設計→テストケース実装→自動実行まで一気通貫で生成

  ```
  文書取り込み（src/ingest Doc Fusion: ルールベース）
    → 要件トレース（req_tracer）
    → MBT モデル構築（networkx 既存グラフ＋文書由来要件のマージ）
    → テスト選択基準の指定（ISTQB CT-MBT 用語: エッジ網羅/頂点網羅/到達目標）
    → パス生成（ルールベース・LLM 不要）
    → テストケース化 → spec.ts 生成 → AutoRun 実行
  ```

- **キー判断**（調査で裏取り済み）:
  - networkx の画面遷移グラフは MBT モデル（有向グラフ）そのもの。GraphWalker の「ジェネレータ×停止条件」設計と ISTQB CT-MBT の語彙だけ借り、**実装は networkx で自前**（LLM 不要・Java 不要）
  - GraphWalker 同梱は Java 依存で venv+systemd 方針と不整合、AltWalker は GPL-3.0 で取り込み不可
  - Chat（対話で要件を練る）は「OpenAI 設定時のみの編集レイヤ」。未設定時は構造化フォーム（取り込んだ画面・項目・要件のテーブル編集）へフォールバック。**パイプライン全段はルールベースで完結**し「OpenAI 未設定でも全機能が動く」原則を守る
- **既存資産**: `src/ingest/`（Doc Fusion・req_tracer）、`src/graph/transition_graph.py`、`web/services/spec_ts_generator.py`、AutoRun
- **用語**: 現行 AutoRun＝「URL駆動」（実測起点）、本モード＝「文書駆動」（要件起点）。実装時に CONTEXT.md へ正式登録

### P1-2. 手動テスト手順書の自動生成（案11） — C

- **内容**: MBT パイプラインのテストパスを、spec.ts だけでなく人間が読める手順書（ステップ番号＋各画面スクショ付き）としても出力。手動実行が多数派の日本市場で、自動化されない案件にも届く
- **主張境界**: パスと画面は実測。期待結果の記述は生成物でありレビュー必須と明記
- **実装ポイント**: P1-1 の出力形式を1つ追加（パス→Markdown/Excel 手順書）

### P1-3. 検収・監査向けテスト実施証跡パック（案4） — C/D

- **内容**: テスト実行結果・スクリーンショット・観点との対応・実行環境情報を検収提出用の1つの報告書に自動整形
- **主張境界**: 「実行した事実の記録」のみで品質判定はしない
- **実装ポイント**: evidence 基盤・`web/services/failure_classifier.py` の出口に整形層を追加

### P1-4. 実測バリデーション仕様（案12） — Q・競合に類例なし

- **内容**: フォームに境界値（101文字・全角・空文字等）を**入力のみ（送信しない）**行い、クライアント側バリデーションの実挙動（エラーメッセージ・入力拒否）を記録。「宣言された制約 vs 実際の挙動」の対応表を仕様書に載せ、実装と仕様の乖離を検出
- **主張境界**: クライアント側で観測できた挙動のみ。送信は一切しない（クロール礼儀の破壊的リクエスト遮断と整合）
- **実装ポイント**: 状態探索（`src/capture/`）と同じ「操作して観測」パターン。テストデータは P1-5 の生成器を流用

### P1-5. テストデータ生成器（案6） — C

- **内容**: 実測したフィールド制約（必須・max_length・型・選択肢）から境界値・同値クラスの具体的入力値セット（99/100/101文字の日本語、全角半角、絵文字等）を CSV/JSON で出力
- **主張境界**: 制約が実測できた項目のみ。各値に導出根拠（どの制約から生成したか）を付ける
- **実装ポイント**: `src/analyzer/test_conditions.py` の制約データを入力に生成層を追加

### P1-6. Drift Check as Code（CI 統合） — D

- **内容**: `--compare --ci` がドリフト検知時に非ゼロ exit code＋JSON サマリを返すモードと、GitHub Actions ワークフローテンプレートを提供。「本番仕様と差分がないこと」を CI ゲートにできる
- **根拠**: Checkly の Monitoring as Code 型
- **実装ポイント**: `src/main.py` に exit code 規約と `--ci` を追加。spec.ts を Actions で回す例も同梱

### P1-7. REST API 拡充と OpenAPI 公開 — C

- **内容**: スケジュール・通知設定の CRUD を API に追加し、`/api/v1/openapi.json` と対話的ドキュメント（Swagger UI 同梱）を公開
- **実装ポイント**: `web/routes/api_v1.py` に schedule/notifier ルートを追加、apispec で仕様生成

### P1-8. 差分の見せ方強化（誤検知フィルタ・重要度） — C

- **内容**: (a) diff レポートに重要度フラグと変更要約、(b) 動的要素（日付・カウンタ）の無視ルール（セレクタ/正規表現）、(c) スクリーンショット比較ビュー。OpenAI 設定時は LLM 要約、未設定時はルールベーススコアリングでフォールバック
- **根拠**: Visualping の AI 要約・IMPORTANT フラグ、Fluxguard の誤検知フィルタ、Octomind の誤検知 78% 削減 KPI
- **実装ポイント**: `src/diff/` に前処理として無視ルールを追加

### P1-9. マルチビューポート仕様書（案8） — Q/C

- **内容**: PC・タブレット・スマホの3ビューポートでクロールし、「スマホでは表示されない項目」「モバイル専用の遷移」をビューポート間差分として文書化
- **主張境界**: 選択したビューポートでの観測のみ
- **実装ポイント**: Playwright の viewport 設定切り替え。差分表示はドリフト検知の比較エンジンを流用

### P1-10. SSO（OIDC 先行）と API トークンスコープ — 企業導入要件

- **内容**: OIDC（Microsoft Entra ID / Google Workspace）ログイン、API トークンに読み取り専用スコープ。SAML は P2
- **実装ポイント**: Authlib（BSD）で OIDC RP。既存の利用者/メンバーシップモデルに載る

### P1-11. 可観測性（メトリクスと構造化ログ）

- **内容**: `/metrics`（Prometheus 形式: クロール成功/失敗数・所要時間・スケジュール遅延・ジョブ滞留・通知成否）と JSON 構造化ログ
- **実装ポイント**: prometheus-client（Apache-2.0）を追加し、scheduler / job_queue / notifier に計測点

### P2-1. API 仕様の逆生成（案7） — C

- **内容**: クロール中に観測した API 呼び出しから「画面↔API 対応表」と OpenAPI 雛形を生成。API 結合テスト支援（mabl 型の「E2E に API を連結」）への足場

### P2-2. 画面カバレッジマップ（案13） — Q

- **内容**: 自動テストが踏んだ画面・遷移を既存の画面遷移図に色で重ね、未踏領域を可視化
- **主張境界**: 「踏んだ範囲」の事実表示に徹する。カバレッジ率を品質保証の数値として掲げない（踏んだ＝検証したではない、と注記）
- **前提**: AutoRun の実行実績が溜まってから価値が出るため P2

### P2-3. 文言一貫性・表記ゆれチェック（案15） — Q/C

- **内容**: 全画面テキストから同義語混在（ログイン/サインイン）・全角半角ゆれ・敬体常体混在を辞書ベースで検出。辞書はワークスペース単位で育てる（組織の用語集が蓄積＝スイッチングコスト）
- **主張境界**: 辞書に登録されたパターンの検出のみ。文章品質の評価はしない

### P2-4. テストレベルの線引き宣言（ユーザー案3の帰結） — 製品方針

- **方針**: 「受け入れ〜システム〜API 結合までを一気通貫、単体テストはスコープ外」と明示的に宣言する
- **根拠**: mabl・testRigor など世界の先行製品も E2E＋API まで。単体はソースリポジトリ接続が必要で製品の性格が変わる（別製品級）。将来は「リポジトリ接続オプション」として再評価

## 実装順序の推奨

1. **第1弾（信頼の底＋即効）**: P0-1 → P0-2 → QW-1 → QW-2 → P0-3
2. **第2弾（運用基盤）**: P0-4 → P0-5 → P1-6（CI 統合は既存 CLI の小改修）
3. **第3弾（差別化の中核）**: P1-1（MBT）→ P1-2（手順書）→ P1-5（テストデータ）→ P1-4（実測バリデーション）
4. **第4弾（商用パッケージ）**: P1-3（証跡パック）→ P1-8（誤検知フィルタ）→ P1-7（OpenAPI）→ P1-9〜11

各弾の着手前に個別設計書（対象ファイル・データモデル・テスト計画）を作成し、レビューを経てから実装モデルへ引き渡す。
