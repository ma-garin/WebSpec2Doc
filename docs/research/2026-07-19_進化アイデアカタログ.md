## B. LLMありのアイデア（設定時のみ有効・既定オフの差し込み口として）

すべて「LLMは候補を出す、確定は人」の2段構えにする。トレーサビリティ研究が
phantom links 対策として人の確認を必須と結論づけているため。

1. **要件⇔画面突合のセマンティック補助** — 現行のルールベース突合で「対応画面なし」となった要件に対し、埋め込み類似度＋LLMで候補画面を提示。確定は人。突合率が上がるほど文書駆動の価値が上がる、最も効く投資先
2. **差分のLLM要約**（P1-8で差し込み口は設計済み） — breaking差分群を「利用者への影響」の文章に要約。ルールベース要約は既に既定で動くため、純粋な上乗せ
3. **観測ベースのテスト観点提案** — 実測したフォーム・遷移・エラーメッセージから「この画面で試すべき観点」を提案し、観点管理DBへドラフト投入（既存の観点承認フローに乗せる）
4. **エラーメッセージの文言品質レビュー** — 実測したバリデーションメッセージ（第3弾で収集済み）に対し「原因が分かるか・次の行動が分かるか」を評価。文言チェック（P2-3）の辞書ベースを補完
5. **MR候補の拡張** — A-4のルールベースMR（3種）をLLMで画面文脈に合わせて拡張（例: 予約系なら「開始日≦終了日の入替は拒否される」）。候補提示のみ
6. **手動手順書の自然文化** — manual_procedures のステップを、対象業務の語彙で読みやすく整形。構造（画面ID・証跡リンク）は保持し文章だけ生成
7. **WebTestPilot型のオラクル推論**【大】 — 要件文書から期待結果そのものを推論して自動判定。研究最前線だが幻覚リスクが主張境界と正面衝突するため、導入するなら「判定はすべて provisional（人の確定待ち）」の第3状態を成果物スキーマに追加してから

## C. LLM不要で次に効くもの（未実装・優先順）

1. **多属性ロケータ候補列（Similo型）** — クロール時実測済みの element_id/aria_label/role/placeholder を spec.ts に候補列として埋め込み、失敗時に「次候補で再試行→成功したら repair 提案として記録」。自動張替はしない（マスキング回避）
2. **性能観測（Core Web Vitals）** — クロール中に LCP/CLS/TTFB を記録するだけで「画面別性能一覧」という新しい成果物になる。Playwright で取得可能・追加コスト極小。主張境界は「測定環境での観測値」
3. **セキュリティヘッダ観測** — 応答ヘッダの CSP/HSTS/X-Frame-Options 等の有無を画面別に一覧化。有無の事実のみ記載（推奨判定はしない）
4. **要件記述テンプレート** — 文書駆動モードのUIに boilerplate 記法ガイドを添付（抽出精度が構造的に向上）
5. **フレイキー切り分けの自動再実行** — async_wait 分類された失敗のみ1回だけ再実行し「再現/非再現」を証跡パックに記録
6. **状態等価のフラグメントベース化（FRAGGEN型）**【大】 — SPA対応を本格化する際の本丸

## D. プロダクト・基盤レベル（技術以外も含む）

1. **WebTestBench での自己評価** — E2Eテスト自動化エージェントの公開ベンチマーク。商用主張の客観根拠になる
2. **成果物の言語切替（英語出力）** — 生成物テンプレートの i18n。海外開発ベンダーとの検収に直結
3. **CLI/CI専用ディストリビューション** — GUI無しの `pip install` 一発構成。Drift Check as Code の普及経路
4. **観点辞書・文言辞書のコミュニティ共有形式** — ワークスペース単位で育てた辞書をエクスポート/インポート可能に（スイッチングコスト源泉の強化）

## 見送り（理由付き）

- リスクベース回帰テスト: ロードマップで不採用決定済み（十分性の主張になる）
- 自動ロケータ張替（完全自動 self-healing）: マスキングリスクが主張境界と衝突
- ソースリポジトリ接続による単体テスト生成: `docs/TEST_LEVEL_POLICY.md` でスコープ外と宣言済み
