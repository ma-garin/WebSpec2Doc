# 機能拡張ロードマップ — 現新比較検証・UX 検証との繋ぎ込み

ステータス: 提案（2026-07-03）。市場調査に基づく開発ロードマップ。

## 0. 出発点

商用テスト検証サービス（ベリサーブ等）のサービスラインである
**「現新比較検証サービス」**（OS 移行・ブラウザ VerUP 時の現行/新システム比較。移行時テストの最大 80% を占めるとされる）と
**「UX・ユーザビリティ検証サービス」**（エキスパートレビュー・ユーザテストによる利用価値向上）を、
既存の開発テーマ 3 件（iframe/Shadow DOM 対応・LLM 観点の要素レベル evidence・残バックログ）と繋ぎ込み、WebSpec2Doc を一段レベルアップさせる。

### 決定的な接続点

現新比較検証の代表的な課題は次の 4 つとされる:

1. 比較の正確性
2. 検証コストの抑止
3. 短納期
4. **設計ドキュメントの不備**

このうち 4 は、WebSpec2Doc の中核機能「URL から仕様書を復元する」がそのまま解決策になる。
画像比較ツール（T-DASH・MagicPod・ATgo 等）は 1〜3 しか扱えない。**「仕様の比較」と「ドキュメント復元」を同時にできる現新比較ツールは調査した範囲で存在しない**。ここが本システムの参入角度になる。

## 1. 市場調査サマリ

### 1-1. 現新比較・マイグレーション市場（追い風が強い）

- 国内 IT モダナイゼーションサービス市場は 2025 年で約 1 兆 3,044 億円（前年比 +10.1%）、CAGR 10.2% で 2030 年に約 2 兆 1,234 億円と予測（IDC）。大企業・中堅企業の約 8 割がレガシーシステムを保有
- 「2025 年の崖」（経産省）: レガシー依存を続けた場合、年間最大 12 兆円の経済損失リスク
- 移行時のテストは工数の最大 80% を占めるとされ、現新比較の自動化ツールが複数商用化されている（T-DASH の画像比較、MagicPod の画像差分、ATgo の AI 画像比較（2024 年に画像内テキスト読取り差分を追加））
- 既存ツールは**ピクセル/画像比較が主軸**。仕様（フォーム構造・バリデーション・画面遷移・カバレッジ）の比較まで踏み込むものはない

### 1-2. ビジュアルリグレッションテスト（技術の参照点）

- Percy / Chromatic / Applitools が 3 強。Applitools は Visual AI による「意味のある差分」抽出、Percy は 2025 年末に AI Visual Review Agent（誤検知の約 4 割を自動フィルタ、レビュー時間 1/3）を投入
- 教訓: **ピクセル差分は誤検知との戦い**。レンダリングゆらぎの吸収（アンチエイリアス・動的コンテンツのマスク）と「意味のある差分」への絞り込みが製品価値の中心。既存の `src/diff/screenshot_diff.py` を現新比較に転用する際は、この誤検知抑制の設計が最重要

### 1-3. AI テストエージェント市場

- AI テストツール市場は 2025 年 6.9 億ドル → 2035 年 38 億ドル予測
- 「AI 支援型」（mabl・Autify・Katalon: 人のワークフローを AI が補助）と「自律型」（testRigor・QA Wolf 等: 意図や観測から AI がカバレッジを自律生成）に二極化
- WebSpec2Doc は「自律探索でテスト設計の**入力（仕様）**を作る」ポジションで、両陣営の上流に位置する（docs/09 の差別化と整合）

### 1-4. UX・アクセシビリティ自動評価

- 自動 a11y 検査の定番は axe-core（MIT ライセンス、DOM があれば動作、WCAG 基準に紐づく構造化違反を返す）。Lighthouse の a11y カテゴリも内部は axe-core
- ただし自動検査で捕捉できるのは a11y 問題全体の 30〜40% → 「ツールによる一次判定＋人の二次判定」がベリサーブ含む業界標準の構え
- LLM によるヒューリスティック評価（ニールセン 10 原則）は研究が進むが限界も明確:
  - GPT-4o の評価は人間専門家の指摘との一致率 21.2% にとどまり、幻覚による誤検知が発生（2025 年比較研究）
  - 30 サイト・850 件超の評価で検出一致率 84%・Cohen's Kappa 0.50（中程度）という報告もあり「早期スクリーニング用途なら実用域」
  - → **evidence-only 原則（セレクタ・bbox・スクリーンショットに紐づかない指摘は出さない）を LLM UX 評価に適用すれば、幻覚問題への直接の対策になる**。これは本システムが既に持つ設計思想の自然な延長
- コードからの仕様復元ツール（EPAM ART、Reversa 等）は登場しているが、いずれも**ソースコードが必要**。URL だけで動く本システムの差別化は維持されている

                    │ Sprint A: iframe/Shadow DOM 対応│ ← 両サービスの土台

              ┌──────────────────┴──────────────────┐

   │ Sprint B: 現新比較モード │              │ Sprint C: UX 自動レビュー │
   │ （現新比較検証サービス対応）│              │ （UX検証サービス対応）      │
   └──────────┬──────────┘              └───────────┬───────────┘

                    │ Sprint D: レポート/ROI 統合    │

- **iframe/Shadow DOM（テーマ1）** は両サービスの前提。現新比較の主戦場であるレガシー業務システムは frameset/iframe 構成が多く、モダン側は Web Components（Shadow DOM）が増えている。どちらも今のクローラは素通りする
- **LLM 観点の要素レベル evidence（テーマ2）** は UX 自動レビューの実装機構そのもの。「どの要素を根拠にした指摘か」を強制することが幻覚対策になる
- **残バックログ（テーマ3）** のうち MutationBlocker 遮断の監査ログ化・カバレッジギャップ提示は、現新比較の「比較の網羅性証明」（どこまで見た/見ていないの明示）として昇格する

## 3. スプリント計画

### Sprint A: iframe/Shadow DOM 対応（土台・約1週間）

- Playwright は open な Shadow DOM を CSS ロケータで自動貫通するため、要素抽出（フォーム・リンク・アクション）の収集 JS を `shadowRoot` 再帰走査に拡張するのが主作業。closed shadow root は「検出したが読めない」として evidence に記録（未確認を未確認と言う）
- iframe は `frame_locator` / `page.frames` で走査し、フレーム内のフォーム・リンク・見出しを親ページの仕様に統合。クロスオリジン iframe は「外部埋め込み」として記録のみ
- 受け入れ条件: デモサイトに iframe ページと Web Components ページを追加し、実ブラウザ E2E でフォーム抽出・状態検出が通ること

### Sprint B: 現新比較モード（約2週間）

ベリサーブの検証フロー（対象システム調査 → 対象画面・帳票整理 → テストケース作成 → テスト実行 → 結果確認）のうち、下線部を自動化する:

1. **2 ターゲットクロール**: 現行 URL と新 URL のペアを受け取り、それぞれをクロール（既存 crawl_urls を 2 回実行＋対応付け）
2. **画面対応付け**: URL パス・タイトル・fingerprint の類似度で現新の画面ペアを自動マッチング（未対応画面は「追加/削除」として報告）
3. **三層比較**:
   - 仕様差分（既存 diff 層を転用: フォーム構造・バリデーション・遷移・画面インベントリ）
   - 画像差分（既存 screenshot_diff を転用。動的領域マスクと閾値設定で誤検知抑制）
   - リンク切れ・HTTP エラー検査（チラシの想定不具合「リンク切れによる操作不可」に対応）
4. **不具合分類レポート**: 表示崩れ / 文字化け・意味消失 / 理解不可能 / 操作不可 の 4 分類（チラシの想定不具合分類に準拠）で報告。全指摘に evidence（現新両方のスクリーンショット・セレクタ）を付与
5. 出力: report.html に「現新比較」タブ、Excel に比較結果シート

差別化: 画像比較ツールと違い、**復元済み仕様書が同時に手に入る**（課題「設計ドキュメントの不備」の解消）。比較は仕様レベルなので「ピクセルは違うが仕様は同じ」（フォント差等）と「ピクセルは同じだが仕様が壊れた」（required 属性消失等）を区別できる。

### Sprint C: UX 自動エキスパートレビュー（約2週間）

「ツール一次判定＋人の二次判定」の一次判定部分を提供する:

1. **a11y 検査（rules 層・confidence 1.0）**: クロール時に axe-core（MIT）を注入実行し、WCAG 違反を画面ごとに evidence（セレクタ・違反ルール ID）付きで記録
2. **ヒューリスティック評価（LLM 層・confidence ≤ 0.9）**: ニールセン 10 原則を観点テンプレート化し、画面のスクリーンショット＋抽出済み要素インベントリを入力に指摘を生成。**要素レベル evidence を必須にし、実在しないセレクタを参照する指摘は破棄**（幻覚フィルタ）。LLM なしでは rules ベースのチェック（ラベル無し入力・コントラスト・タップ領域サイズ等）にフォールバック
3. 出力: report.html に「UX 所見」タブ（重大度 × 画面のマトリクス）、確信度と根拠を明示し「自動検査は a11y 問題の 30〜40% を捕捉する一次スクリーニングである」旨を免責表示（evidence-only 原則）

### Sprint D: レポート/ROI 統合（約1週間）

- usage_tracker に現新比較・UX レビューの実績を追加（比較画面数・検出差分数・UX 指摘数 → 削減工数の係数化）
- 残バックログ消化: MutationBlocker 遮断の監査ログ化、カバレッジギャップ（未探索領域）の明示をレポートに追加 → 現新比較の網羅性証明として表示
- デモ台本に「移行検証デモ」（DemoMart の新旧 2 バージョンを比較）と「UX 診断デモ」を追加

### 推奨の優先順位

1. **§6 の文書×実測融合（Doc Fusion）**: 製品の性格を「復元ツール」から「突合・再生ツール」へ変える最重要軸（詳細は §6）
2. **B3 境界値テストデータ生成 + B2 項目定義書**: 既存の実測バリデーション資産の直接延長で、1 スプリントで作れて成果物の説得力が大きい。4 スプリント計画の Sprint D に合流可能
3. **C1 MCP サーバー化**: 「AI エージェント時代のレガシー仕様供給源」という位置づけ。Flask API が既にあるため薄いラッパーで PoC 可能
4. **A1 認証レコーダー + B1 API 仕様復元**: 実案件投入で最初にぶつかる壁と、リプレイス案件での追加価値。実戦配備の直前に実装するのが効率的

## 6. 文書×実測の融合（Doc Fusion）— 最重要の進化軸

### 6-1. 発想の転換

現在のシステムは「ドキュメントが何もない現場」を前提に URL だけから仕様書を復元する。しかし実際の現場の多数派は「**文書はあるが古い・信用できない**」状態である（RFP・企画書・基本設計書・画面一覧・項目定義書が数世代前のまま放置されている）。

古い文書は「間違った情報」ではなく「**劣化した貴重な情報源**」と捉え直す:

- 文書には**画面からは絶対に読み取れない情報**がある: 正式な画面名・帳票名、業務用語、業務ルールの意図、画面 ID 体系、権限マトリクス、非機能要件
- 実測には**文書が持ち得ない情報**がある: 今この瞬間の本当の挙動、実際のバリデーションメッセージ、実際の遷移
- 両者を突合すれば、単独では作れない品質の仕様書と、**「文書と実装のズレ」という新しい成果物**が生まれる

### 6-2. 5 つの機能

| 機能 | 内容 | 成果物 |
|---|---|---|
| ① ギャップ分析 | 文書記載の画面・項目・ルールと実測を突合し、「文書のみ（実装から消えた/未実装）」「実装のみ（文書化漏れ）」「両方あるが矛盾（桁数・必須・遷移が違う）」の 3 分類で報告 | ギャップレポート（既存ドリフト検出の一般化: crawl vs crawl → doc vs crawl） |
| ② 用語・文脈の注入 | 文書から正式名称・業務用語・画面 ID 体系を抽出し、生成仕様書の画面名・項目名に反映（「P001」→「与信申込入力画面（GA-010）」） | 「その会社の文書」として通用する仕様書 |
| ③ 要件トレーサビリティ | RFP・要件一覧の各要件を対応画面・機能・テストケースへマッピング（evidence 付き）。対応先が見つからない要件は「未実装疑い」として提示 | 要件カバレッジマトリクス（受入テストの基盤） |
| ④ テスト観点の高度化 | 画面からは読めない業務ルール（計算式・限度値・権限条件）を文書から抽出し、対応画面のテスト観点・境界値に注入 | 業務ルール由来のテストケース（B3 境界値生成と直結） |
| ⑤ 文書の再生（リフレッシュ） | 古い仕様書を実測に合わせて改訂した新版を生成。変更箇所には「実測により更新（旧: 全角 20 桁 → 実測: 40 桁）」の注釈 | 更新済み仕様書 — 「復元」より顧客の心理的受容が高い（ゼロから作り直しではなく、自分たちの資産が蘇る） |

### 6-3. evidence-only 原則の拡張

根拠が 2 系統になる。SourceEvidence を拡張し、すべての記述にどちら由来かを明示する:

- **実測 evidence**（既存）: selector・bbox・スクリーンショット。confidence 1.0
- **文書 evidence**（新規）: ファイル名・ページ/セクション・原文引用。confidence は文書の鮮度と抽出方式に応じて設定（LLM 抽出は ≤0.9）
- **矛盾時は隠さず両方提示**（「文書では必須、実測では任意」）— この矛盾リスト自体が顧客への最初の納品価値になる

### 6-4. 実装スケッチ

```text
新層 ingest/（文書取り込み）
  Excel（openpyxl・既存依存）→ 画面一覧・項目定義書のパース
  PDF/Word → テキスト抽出 → LLM Structured Outputs で画面・項目・ルールへ正規化
  ↓ 正規化モデル: DocumentedScreen / DocumentedField / DocumentedRule（文書 evidence 付き）
analyzer/ に突合器（matcher）
  画面名類似・URL・項目名・ラベルの類似度で文書側と実測側をマッチング
  （現新比較モードの「画面対応付け」と同一機構 — Sprint B と実装を共有できる）
generator/
  統合仕様書（用語注入済み）＋ギャップレポート＋トレーサビリティマトリクス
web/
  文書アップロード UI（クロール設定画面に「参考文書」欄を追加）
```

既存資産との整合が非常に良い: 突合器は Sprint B（現新比較）の画面マッチングと共通、ルール注入は B3（境界値生成）の入力になり、LLM 層は既存の Structured Outputs 基盤をそのまま使う。

### 6-5. 競合状況と市場空白

- 調査した範囲で、コード解析型（EPAM ART・Reversa）は文書を扱わず、文書 AI（RAG 型の文書検索・Q&A）は実装と突合しない。**「文書 × 生きているシステムの実測」の突合は空白地帯**
- サービスメニューとして「**仕様書再生サービス**」（お手元の古い設計書一式＋URL → 更新済み設計書＋ズレ一覧）が成立する。「仕様書復元」（白紙前提）より案件の入口が広い — 文書が全く無い現場より、古い文書を抱えた現場の方が多い
- 現新比較（Sprint B）とも連結する: 移行前システムの旧文書を取り込めば、現新比較の期待値ソースが「現行システムの実測」と「文書」の 2 つになり、検証の説得力が上がる

### 6-6. 段階的な作り方

1. **Phase 1（1 スプリント）**: Excel 画面一覧・項目定義書の取り込み → 用語注入（②）とギャップレポート（①）の項目レベルのみ。LLM 不要（ルールベース突合）で確実に動く
2. **Phase 2（1 スプリント）**: PDF/Word + LLM 抽出、業務ルール注入（④）、文書 evidence の UI 表示
3. **Phase 3（1 スプリント）**: トレーサビリティマトリクス（③）と文書再生（⑤）

## 7. プロジェクト視点の進化軸 — テスト実行ライフサイクルへの拡張

### 7-1. 視座の転換

これまでの機能はすべて「URL（＋文書）を入れたら何が出てくるか」というプロダクト視点だった。しかし QA プロジェクトの全体（計画 → 設計 → 実行 → 報告 → 保守）で見ると、本システムは**上流（設計インプット作成）で降りてしまい、実行以降を支えていない**。

プロジェクト視点で見たときの本システム固有の強みは一つに集約される:

> **クローラは「分母」を持っている。**
> テスト管理ツールは「何をテストしたか（分子）」しか知らない。
> 本システムは全画面 × 全状態 × 全機能のインベントリ＝分母を機械的に構築できる。
> 分母を持つ者だけが「テストされていない場所」を可視化できる。

### 7-2. キャプチャ & リプレイ＋カバレッジヒートマップ（中核機能）

テスターやユーザーに実際のシステムを操作してもらい（ハッピーパスの実演・探索的テスト）、その操作からテスト資産を逆生成し、探索の足跡をインベントリに重ねて可視化する。

**① キャプチャ（探索セッション記録）**

- 記録用ブラウザ（Playwright 起動のブラウザ、または CDP 接続）でテスターが自由に操作
- クリック・入力・遷移・出現した状態（モーダル等）をセッションログとして記録（セレクタ・スクリーンショット・タイムスタンプ付き = 既存 evidence モデルをそのまま利用）
- 気づき登録: 操作中にワンクリックで「バグ疑い」「気になる」をマーク → 再現手順は操作記録から自動生成

**② リバース（操作からテスト資産を逆生成）**

- 操作列 → テストケース化: 「操作手順（実測）＋期待結果（観察された結果）」の形式で、既存のシナリオ生成と同じスキーマに落とす
- 操作列 → 業務フロー化: 実演されたハッピーパスを遷移グラフ上のパスとして登録し、ビジネスフロー優先度付けの入力にする（現在はヒューリスティック推定 → 実演由来なら confidence 1.0）
- Scribe/Tango 等の録画→手順書ツールはテスト資産まで逆生成しない（§5 A2 の調査済みギャップと同一。A2 はこの機能に統合する）

**③ ヒートマップ（探索カバレッジの可視化）**

- **接合キーは fingerprint**: 記録された操作時の画面状態を、クロール済みインベントリの page_id / state fingerprint（実装済みの fingerprint v2）と照合 → どの画面・状態・フォーム・アクションが「触られたか」をマッピング
- report.html に画面 × 機能のヒートマップタブ: 濃い = 何度も探索された / 薄い = 一度だけ / 白 = **未探索**。状態レベル（モーダル・タブ）まで塗り分ける
- 「未探索 × ビジネスフロー優先度・高」のセルを次の探索チャーターとして自動提案 → 探索的テストのセッションベース管理（SBTM）が回り出す

**④ リプレイ（記録の自動リグレッション化）**

- 記録済みハッピーパスを再実行可能にし、夜間リプレイ = 実利用由来のリグレッションスイート（AutoRun 基盤を転用、セレクタは evidence から自動修復候補を提示）
- ドリフト検出と連結: 差分が出た画面を通過する記録済みフローを特定し「先に再実行すべきフロー」として提案（既存の差分影響分析を記録フローへ拡張）

### 7-3. プロジェクトライフサイクル対応表

| フェーズ | 現状 | 追加するもの |
|---|---|---|
| 計画 | —（対象外） | インベントリ＋ROI 係数からテスト計画ドラフト生成（画面数 × 優先度 → 工数見積・スコープ表） |
| 設計 | ◎ 仕様書・テストケース生成（中核） | 文書融合（§6）・境界値データ（B3）で強化 |
| 実行 | △ AutoRun のみ | キャプチャ & リプレイ（§7-2）・気づき→バグ票自動起票（再現手順・環境・スクリーンショット付きの社内フォーマットで出力） |
| 報告 | △ ROI ダッシュボードのみ | カバレッジヒートマップ・消化バーンダウン（分母に対する進捗）・ステークホルダー向け日次サマリ |
| 保守 | ○ ドリフト検出 | ドリフト × 記録フローの影響提案・CI 監視（C4）・リプレイによる回帰確認 |
| 引き継ぎ | —（対象外） | 仕様書＋記録済みフロー再生 = 新規参画者のドメイン知識教材（オンボーディングパック） |

### 7-4. 補助アイデア

- **アクセスログによる重み付け**: GA・サーバログを任意入力として受け取り、実ユーザーのトラフィックで画面に重みを付ける → 「ユーザーが実際に通る場所からテストする」リスクベース優先度の実データ化
- **チーム探索の統合**: 複数テスターのセッションを集約し、チーム全体のヒートマップと個人別チャーター配分を表示（SBTM の運営画面）

### 7-5. 実装の足がかり

必要な部品の大半が既にある。(a) 操作記録 = Playwright の既存基盤＋evidence モデル、(b) 照合 = fingerprint v2、(c) 可視化 = report.html のタブ機構、(d) リプレイ = AutoRun、(e) バグ票 = 既存 report.html の evidence 情報をそのまま転記するテンプレート出力。**新規に必要なのは「セッション」という概念（記録・保存・集計のモデル）と記録用 UI のみ**。Phase 1 として「キャプチャ＋ヒートマップ」（リバース・リプレイは後続）なら 1〜2 スプリントで成立する。

この機能があると、デモの締めも「仕様書ができました」から「**テストの進み具合と穴が、この地図で毎日見えます**」へと語り口が変わる。
