# 自動ログイン（GUI内フォーム入力方式）に切り替える

**ADR-0001を廃止し、本ADRに置き換える。**

## 文脈

ADR-0001 では「手渡しログイン（別ウィンドウ＋シグナルファイル）」を採用し、自動ID/PW入力を棄却した。理由は MFA/SSO/CAPTCHA 対応とパスワード非保持の思想だった。

しかし実際の操作フローを検討した結果、別ウィンドウが開くことによる操作性の低下（GUI との文脈断絶）が実務上の障壁として大きすぎると判断した。対象サイトはID/PASSで認証できる社内システムが主であり、MFA/SSO対応は現フェーズのスコープ外とする。

問題は「パスワードをツールが保持すること」ではなく「パスワードをどう扱うか」である。送信後に即破棄し、セッション（Cookie）のみを保存する設計であれば許容できる。

## 決定

**手渡しログインを廃止し、GUI内フォーム入力による自動ログインを採用する。**

### フロー

1. ページ解析（`/api/discover`）で `login_required: true` なページを検出する
2. 検出したリダイレクト先URL（ログインページ）をGUIのログイン設定欄に自動セットする
3. ログインページのフォームフィールドを動的スクレイプし、GUI上に表示する
4. ユーザーがGUI上でID・パスワード等を入力し、送信する
5. ツールがフォームを自動送信し、`detect_login_wall()` で成功を判定する
6. MFA等の追加フィールドが出現した場合、再スクレイプしてGUIに動的追加する（ステップ4へ戻る）
7. ログイン成功後、入力値を即メモリから破棄し、セッションを `output/{domain}/auth.json` に保存する
8. 認証付き再解析（discover with auth）を自動で走らせ、結果をGUIに表示する

### セキュリティ上の約束

- ID・パスワードはフォーム送信にのみ使用し、保存・ログ出力を一切しない
- `auth.json` にはセッション（Cookie等）のみを保存する。パスワード本体を含まない
- MFA/SSO/CAPTCHAは対応範囲外とし、GUIでその旨を明示する

## 棄却した代替案

- **手渡しログイン（ADR-0001方式）の継続**：MFA/SSO対応は優れるが、別ウィンドウによる操作性低下が実務障壁。対象サイトのほとんどがシンプルなID/PASSログインであり、この制約を払うメリットが薄い。
- **パスワードをローカルに暗号化保存して自動再ログイン**：セッション切れ時の再認証を自動化できるが、暗号化鍵管理のリスクとコストが高い。「再入力を促す」で十分と判断。

## 帰結

- `src/crawler/auth.py` の `capture_auth_state` / `capture_auth_state_via_signal` は廃止
- `web/routes/login.py` の `/api/login/start` / `/api/login/finish` はログインフォームスクレイプ・自動送信・成功判定のエンドポイントに置き換える
- MFAが必要なサイトはGUIで「対応外」と表示して処理を停止する
- セッション保存先（`output/{domain}/auth.json`）は変更なし
