## 1. エグゼクティブサマリ

WebSpec2Doc は「クロール→仕様文書生成→ドリフト検知」の基本ループに加え、スケジューラ・通知・REST API・マルチテナント・監査ログ（クロール理由）まで実装済みであり、**機能の「有無」では変更検知系競合と既に肩を並べている**。ギャップは実装の**成熟度**にある: 通知チャネルの幅とテンプレート性（competitor は apprise 経由で数十チャネル＋Jinja2）、スケジュールの柔軟性（曜日・時間帯ウィンドウ）、失敗時のリトライ・通知、API のカバレッジと OpenAPI 公開、可観測性、保持ポリシーが主な差分。一方、**QA テスト設計インプットの自動生成＋ISTQB 準拠**という組み合わせは調査した範囲でどの競合も持っておらず、ここに networkx の画面遷移グラフを活かした**ルールベース MBT（モデルベーステスト）生成**を重ねることが最も守りやすい差別化になる。実務定着の観点では、オンボーディングツアー（MIT ライセンスの driver.js をローカル同梱）と CI 統合（drift check as code）が費用対効果の高い一手。

## 3. 隣接競合（QA自動化系）の要点

| 製品 | 中核アプローチ | WebSpec2Doc への示唆 |
|---|---|---|
| **mabl** | 低コードのAIテスト作成。自動修復（auto-healing）は要素属性・視覚コンテキスト・DOM 位置・周辺構造の複数シグナルを同時に使いロケータ破損に耐える（保守 85% 削減を訴求）。API テストを E2E フローに連結（状態セットアップ／バックエンド検証） | Playwright 候補生成に**多重ロケータ（evidence を複数保持）**を持たせると再クロール後のテスト耐性が上がる |
| **testRigor** | テストを平易な英語で記述し、実行のたびにページ状態に対して再解釈（ロケータ問題を回避）。CI/CD 連携 | 自然言語→テストの流れは「文書駆動モード」（§5 P1-4）の先行事例 |
| **QA Wolf** | サービス型。数ヶ月で E2E カバレッジ 80%・QA サイクル 15 分を訴求 | 「カバレッジ％」を顧客への約束として掲げる打ち出し方 |
| **Octomind** | 自動 E2E 生成＋自動修復。誤検知 78% 削減を訴求 | 誤検知率は信頼性の KPI として明示する価値がある |
| **Meticulous**（二次情報） | 実ユーザーセッションを記録・リプレイして回帰検知（DOM 変異・JS イベント・ネットワークを計測） | トラフィックが既にあるサイト向けの回帰検知アプローチ。将来のドリフト検知精度向上の参考 |
| **Checkly** | **Monitoring as Code**: 監視定義を標準 `*.spec.ts`（Playwright、ロックインなし）＋設定ファイルとしてリポジトリに置き、CLI/Terraform/GitHub Actions から test → deploy する | WebSpec2Doc の spec.ts 生成と直結する運用モデル。「ドリフト検知を CI のゲートにする」提案（§5 P1-1）の根拠 |

**テストレベルのカバー範囲**（ユーザー施策案3の裏取り）:

- mabl・testRigor とも中核は **E2E（システムテストレベル）＋ API テスト**。mabl の API テストは「E2E フローの前後に API 呼び出しを連結する」形であり、テストピラミッドの下層（単体）は扱わない。
- テストピラミッドの一般論（単体多数・結合少数・E2E 最少）はこれらのベンダー自身が解説しているが、**単体テストの生成・実行を提供するブラックボックス型（URLだけ渡す型）製品は調査範囲で確認できなかった**。単体テストはコード（関数・クラス）へのアクセスが前提であり、URL ベースの外形監視とは原理的に情報源が異なるため（この段落の結論部は調査に基づく推論）。

## 5. 施策提案

優先度: **P0 = 実務投入に必須 / P1 = 商用競争力 / P2 = 将来**。

### P0: 実務投入に必須

#### P0-1. スケジューラの運用成熟化

- **内容**: (a) 実行時間帯・曜日ウィンドウ（「平日 2:00-5:00 のみ」等）、(b) 失敗時リトライ（回数上限＋バックオフ）、(c) **クロール失敗の通知**（現状はドリフト時のみ通知でサイレント失敗する）、(d) 実行履歴の永続化（schedule.json とは別に history を記録）と UI 表示。
- **根拠**: changedetection.io（OSS・無料）ですらタイムゾーン＋曜日/時間帯制限を標準装備。監視系製品は「実行されなかったことに気づける」ことが信頼の底線（後半は推測）。
- **適用**: `scheduler.py` の `_calc_next_run_at` をウィンドウ対応に拡張し、`_run_crawl` の失敗パスで `notifier.py` を呼ぶ。cron 式のフル対応は不要（daily/weekly/monthly＋時間帯ウィンドウで実務は足りる—推測）。

#### P0-2. 通知の実務化

- **内容**: (a) Microsoft Teams 対応（Incoming Webhook/Power Automate）、(b) 通知文面のテンプレートカスタム（Jinja2）、(c) diff サマリ（追加/削除ページ名の上位 N 件）を本文へ埋め込み、(d) 「テスト送信」ボタン、(e) クロール失敗通知（P0-1(c) と共通）。
- **根拠**: changedetection.io は apprise + Jinja2 テンプレートが標準。Fluxguard はスクショ・色分け diff をメール本文に直接埋め込み「開かなくても分かる」ことを売りにする。
- **適用**: `notifier.py` の `_build_text` をテンプレート化し、`NotifierConfig` に `template` を追加。apprise（BSD-2-Clause）を依存に加えれば数十チャネルへ一気に拡張できるが、まず Teams＋テンプレートの自前実装で十分（推測）。

#### P0-3. 初回オンボーディングツアー（ユーザー施策案1）

- **内容**: 初回ログイン時に「STEP1: ここをクリック」型のガイド付きツアーを表示。構成は (a) ウィザード4ステップを 3〜5 個のツールチップで巡るツアー、(b) 「最初のサイトを登録する→初回クロール→レポートを見る」のチェックリスト、(c) サイト未登録時の empty state に同梱デモサイトへの導線。
- **ライブラリ選定（ライセンスが決定打）**:

  | ライブラリ | ライセンス | 商用利用 | 備考 |
  |---|---|---|---|
  | **driver.js** | **MIT** | ○ 無償で可 | 約5KB gzip・依存ゼロ・vanilla TS・26.3k stars。npm/ローカル同梱可（CDN 不要） |
  | Shepherd.js | AGPL-3.0＋商用ライセンス | × 商用は有償ライセンス必須（会社が収益を上げていれば対象） | 不採用 |
  | Intro.js | AGPL＋商用ライセンス | × 同上 | 不採用 |

  → **driver.js 一択**。本プロジェクトの「CDN 不可・ローカル同梱」前提も依存ゼロ・5KB で満たす。
- **設計パターンの根拠**: Appcues・Pendo の一次資料より — ツアーは 3〜5 ツールチップ・1 つ 140 字以内・「操作方法」でなく「価値」を語る。Orient（ようこそ＋ツアー）→ Activate（チェックリストで最初の成功体験）→ Reinforce（empty state・完了の称賛）の3段階。初日に詰め込みすぎるのが最頻の失敗。
- **適用**: `static/` に driver.js を同梱し、利用者レコードに `tour_completed` フラグを持たせる。チェックリストの「最初の成功」はデモサイト（`make demo`）クロールに誘導すれば外部依存なしで完結する。

#### P0-4. データ保持ポリシーとバックアップ手順

- **内容**: (a) スナップショット世代管理（保持世代数 or 期間を設定し自動削除、既定は無制限のまま）、(b) ディスク使用量の表示、(c) バックアップ/リストア手順書（output/ と instance/ の対）。
- **根拠**: Wachete は 12ヶ月保持、Fluxguard は永久アーカイブと、**保持期間は製品仕様として明示するのが通例**。EnterpriseReady はデータポータビリティを要求。
- **適用**: スケジューラループに世代 GC を追加。削除は監査ログに記録する。

#### P0-5. 管理操作の監査ログ

- **内容**: audit.jsonl（クロール理由）に加え、ログイン成否・メンバー/ロール変更・スケジュール/通知設定変更・エクスポートをテナント単位で記録し、admin が閲覧できるビューを提供。
- **根拠**: EnterpriseReady「Audit Logs: 管理者にアカウント活動の詳細な証跡を提供する」。SOC 2 / ISO 27001 いずれの統制でも活動証跡は基礎要件。
- **適用**: 既存の audit.jsonl と同形式（JSONL）で `instance/tenants/{slug}/admin_audit.jsonl` に追記。

### P1: 商用競争力

#### P1-1. Drift Check as Code（CI 統合）

- **内容**: `python src/main.py --url ... --compare --ci` がドリフト検知時に非ゼロ exit code を返すモードと、GitHub Actions のワークフローテンプレートを提供。リリース前に「本番仕様と差分がないこと」を CI ゲートにできる。
- **根拠**: Checkly は監視定義をリポジトリに置き CLI で test→deploy する Monitoring as Code を製品の中核に据え、CI/CD からのチェック実行を標準ワークフローとして文書化している。
- **適用**: CLI は既存。exit code 規約と `--ci`（JSON サマリを stdout）を足すだけで成立する。生成済み spec.ts（AutoRun）を Actions で回す例も同梱すると Checkly 型の運用が完全に再現できる。

#### P1-2. REST API 拡充と OpenAPI 公開

- **内容**: スケジュール・通知設定の CRUD を API に追加し、OpenAPI 仕様を `/api/v1/openapi.json` で公開、対話的ドキュメント（Swagger UI 同梱）を付ける。
- **根拠**: changedetection.io は OSS ながらフル REST API＋OpenAPI＋対話的ドキュメントを提供し、SDK 生成まで謳う。Visualping は Free プランから API を開放。
- **適用**: `api_v1.py` に schedule/notifier ルートを追加。Flask なら apispec 等で既存ルートから仕様を生成できる。

#### P1-3. 差分の見せ方強化（AI 要約・誤検知フィルタ）

- **内容**: (a) diff レポートに「重要度」フラグと変更要約を付与、(b) 動的要素（日付・カウンタ等）の無視ルール（セレクタ/正規表現）、(c) スクリーンショット比較ビュー。OpenAI 設定時は LLM 要約、未設定時はルールベース（変更種別・件数・影響画面数によるスコアリング）でフォールバック。
- **根拠**: Visualping は AI 要約と IMPORTANT フラグを全プラン全アラートに付け、これを競合との差別化と明言。Fluxguard は AI 誤検知フィルタが中核機能。Octomind は誤検知 78% 削減を KPI として訴求。
- **適用**: 既存の evidence-only 原則と親和的。無視ルールは diff 層（`src/diff/`）に前処理として追加。

#### P1-4. 文書駆動モード＋ルールベース MBT パイプライン（ユーザー施策案2）

- **内容**: 仕様書・要件文書を取り込み、テスト方針→テスト基本設計→テスト詳細設計→テストケース実装→自動実行まで一気通貫で生成するパイプライン。

  ```
  文書取り込み(src/ingest Doc Fusion: ルールベース)
    → 要件トレース(req_tracer)
    → MBTモデル構築(networkx 既存の画面遷移グラフ ＋ 文書由来の要件をマージ)
    → テスト選択基準の指定(ISTQB CT-MBT 用語: エッジカバレッジ / 頂点カバレッジ / 到達目標)
    → パス生成(ルールベース・LLM不要)
    → テストケース化 → spec.ts 生成 → AutoRun 実行
  ```

- **MBT ツールの一次情報と適合性評価**:
  - **GraphWalker**（MIT・Java）: モデル＝有向グラフ（頂点=状態、辺=遷移）。パス生成は「ジェネレータ（random / a_star 等）×停止条件（edge_coverage(100) / vertex_coverage / reached_vertex / time_duration）」の組み合わせで表現する。CLI jar / REST サービスとして実行。
  - **AltWalker**（**GPL-3.0**・Python）: GraphWalker を使ってパス生成し、頂点・辺を Python クラスのメソッドへマップして実行。unittest 型フィクスチャ（setUpRun/setUpModel）を持つ。
  - **適合性の結論**: WebSpec2Doc は**MBT モデルに相当する有向グラフ（networkx の画面遷移グラフ）を既に構築済み**であり、GraphWalker の「ジェネレータ×停止条件」はグラフアルゴリズム（networkx で実装可能）なので、**LLM にもJavaにも依存せずルールベースで自前実装できる**。GraphWalker 同梱は Java ランタイムが必要で venv+systemd 方針と相性が悪く、AltWalker は GPL-3.0 のため取り込み不可。→ **語彙と設計（ジェネレータ×停止条件、ISTQB CT-MBT のテスト選択基準・カバレッジ基準）だけを借り、実装は networkx で自前**が最適（この結論は調査に基づく評価）。
  - ISTQB CT-MBT シラバス（v1.1）は MBT モデリング・テスト選択基準（カバレッジベース含む6ファミリー）・MBT テスト生成を体系化しており、**日本の QA 市場に対して「ISTQB CT-MBT 準拠の用語で出力する」ことがそのまま差別化になる**（既存の ISTQB 準拠路線とも一貫）。
- **Chat（対話で要件を練る）の扱い**: Chat は本質的に LLM 依存のため、**「OpenAI 設定時のみ有効になる編集レイヤ」**として実装し、未設定時は構造化フォーム（取り込んだ文書の画面・項目・要件をテーブル編集）へフォールバックする。パイプラインの他の全段（取り込み・モデル構築・パス生成・spec.ts 生成・実行）はルールベースで完結するため、「OpenAI 未設定でも全機能が動く」原則を破らない。
- **根拠**: testRigor が「自然言語→テスト」の市場性を実証済み。GraphWalker / AltWalker が「有向グラフ＋カバレッジ停止条件」という実装形を実証済み。既存資産（src/ingest の Doc Fusion、req_tracer、networkx グラフ、spec.ts 生成、AutoRun）で工程の大半が埋まっている。

#### P1-5. SSO（OIDC 先行）と API トークンスコープ

- **内容**: OIDC（Microsoft Entra ID / Google Workspace）でのログインを追加。API トークンに読み取り専用スコープを導入。SAML は P2 へ。
- **根拠**: EnterpriseReady「SSO: 企業が単一の中央ディレクトリからユーザーを管理できるようにする」— 企業導入の最初の関門。
- **適用**: Flask なら Authlib（BSD）で OIDC RP を実装。既存の利用者/メンバーシップモデルにそのまま載る。

#### P1-6. 可観測性（メトリクスと構造化ログ）

- **内容**: `/metrics`（Prometheus 形式: クロール成功/失敗数、所要時間、スケジュール遅延、ジョブキュー滞留、通知成否）と JSON 構造化ログ。
- **根拠**: EnterpriseReady の Reporting & Analytics。監視製品は自身の稼働証明が信頼の前提（推測）。
- **適用**: prometheus-client（Apache-2.0）を追加し、scheduler / job_queue / notifier に計測点を置く。

### P2: 将来

#### P2-1. テストレベル一気通貫の拡張（ユーザー施策案3）

- **評価**: 現在の WebSpec2Doc は**システムテスト（E2E/UI）レベル**のインプット生成・実行支援であり、mabl / testRigor など世界の先行製品も **E2E＋API テストまで**が守備範囲。単体テストを提供するブラックボックス型製品は確認できなかった。
- **結合（API）テスト支援 — 現実的（P1〜P2）**: クロール時に観測した API 呼び出し（既存の api_changes 検知が下地）から API テスト候補を生成する路線は、mabl の「E2E に API を連結する」型と同じで拡張として自然。
- **単体テスト支援 — 高コストのため当面スコープ外を推奨**: URL ベースの外形観測ではコード（関数・クラス）に到達できず、原理的にソースリポジトリ接続が必要。接続するなら言語ごとのパーサ・ビルド/実行環境・依存解決・CI 連携が必要で、製品の性格が「URL を渡すだけ」から「開発環境に組み込むツール」へ変わる（工数・保守とも別製品級 — 評価としての推測）。**「受け入れ〜システム〜API 結合までを一気通貫、単体はスコープ外」と明示的に線を引く**ことを推奨。将来やるなら「リポジトリ接続オプション」として P2 以降で再評価。

#### P2-2. SAML / SCIM

- EnterpriseReady のフルチェックリスト対応。大企業向け商談が現実化した時点で着手（OIDC で大半の日本企業はカバーできる—推測）。

#### P2-3. エディション設計と認証基準

- **内容**: OSS コア＋商用エディション（EnterpriseReady「Product Assortment」）。価格相場は changedetection.io $8.99/月〜 Visualping Solutions $3,000/年〜 Fluxguard $499/月と幅広く、「チーム機能・保持期間・SSO・サポート SLA」で階段を作るのが通例。商用 SaaS 化する場合、日本市場は ISO 27001（ISMS）が既知の言語であり、北米に売るなら SOC 2 Type 2。いずれも管理体制の構築が主工数（数十〜150 統制）。
- 判断時期: 商用提供形態（社内ツール配布か SaaS か）が固まってから。

#### P2-4. 完全アーカイブ（コンプライアンス用途）

- Fluxguard は「全サイトの全編集履歴を永久保存」を規制コンプライアンス・同意取得監視の用途で販売している。金融・製薬など規制業種向けの将来オプション。P0-4 の保持ポリシー基盤の上に「無期限保持＋証跡エクスポート」を載せる形。

#### P2-5. 監視タイプの拡張

- Hexowatch は 13 監視タイプ（キーワード・技術スタック・WHOIS・sitemap 等）、Wachete は PDF/Word/Excel/JSON 文書監視を提供。WebSpec2Doc は QA 文脈に絞るべきで全部は不要だが、**sitemap 監視（新ページの早期検知）と PDF（仕様書・利用規約）監視**は QA ユースケースに重なる（選定は推測）。
