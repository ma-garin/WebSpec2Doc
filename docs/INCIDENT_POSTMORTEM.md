## 1. インシデント概要

AutoRun 承認モーダルの UI を実装・コミット・プッシュした後、ユーザーがスクリーンショットを提供したことで、30件以上の UI/UX 問題が発覚した。

**核心的な問題**: ブラウザによる実動確認（L3 システムテスト・L4 受け入れテスト）を一切実施せずにコードをリリースした。

## 3. 根本原因分析（5 Whys）

**Why 1**: なぜ問題のある UI をリリースしたか？  
→ ブラウザで実際に確認しなかった。

**Why 2**: なぜブラウザ確認をしなかったか？  
→ `pytest` 483本が通過した時点で「完了」と判断した。

**Why 3**: なぜ pytest 通過を完了の基準にしたか？  
→ L3（システムテスト）・L4（受け入れテスト）が必要なことを知りながら、機械的強制力がなかったため省略できた。

**Why 4**: なぜ機械的強制力がなかったか？  
→ AGENTS.md に 10 ステップのブラウザ確認フローが定義されていたが、git hook 等のハーネスに組み込まれていなかった。プロセスは「知っている」が「実行を強制される」状態ではなかった。

**Why 5**: なぜハーネスに組み込まれていなかったか？  
→ 初期設計時に「人間の判断に依存する手順」として定義され、自動化の対象から外れていた。プロセス定義と実行強制が分離した状態が放置された。

**根本原因**:  
プロセスは文書に定義されていたが、機械的な強制力がなく「速度のために省略可能」な状態になっていた。

## 6. 学習事項

### 学習 1: テストレベルの全段階は必須
L1(Unit) + L2(Integration) だけでは「ユーザーが価値を受け取れる状態」を証明できない。  
L3(System) と L4(Acceptance) が揃って初めて「品質が保証された」といえる。

### 学習 2: 「速さ」と「品質」のトレードオフは偽物
ブラウザ確認に5分をかけなかった結果、インシデント対応に2時間以上かかった。  
プロセスを守る方が、長期的には速い。

### 学習 3: プロセスは機械的に強制されなければ機能しない
「知っている」プロセスと「実行が強制される」プロセスは別物。  
ハーネス（hook・CI・自動チェック）なしのプロセス文書は、存在しないも同然。

### 学習 4: QA ツールを作るチームは最高水準の QA を実践しなければならない
このプロダクトは QA エンジニアのためのツールである。  
自分たちが QA を実践していなければ、ツールへの信頼は成立しない。

## 7. 再発防止の検証方法

以下が全て TRUE になったとき、再発防止措置が有効であると判断する:

- `make verify-ui` を実行せずに HTML/JS/CSS 変更をコミットしようとすると **ブロックされる**
- `make verify-ui` の実行により **スクリーンショットが保存される**
- E2E テストが 1 件でも FAIL すると **`.ui-verified` が生成されない**
- 次回の UI 変更時に **このプロセスが守られる**
