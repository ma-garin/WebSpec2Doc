# アプリ利用者ログイン（メール自己申告）とワークスペースを導入する

> **注（2026-07-16 追記）**: 本 ADR で導入した軽量認証は、商用/共有サーバ向けの
> 認証・テナント分離実装（`docs/AUTH_TENANCY.md`、機能 `account_auth`/`tenant_isolation`）に
> 統合・置換された。本書は設計判断の履歴として保持する。

## 文脈

WebSpec2Doc は元来ローカル単一利用者向けのツールで、アプリ利用者の認証・アカウント・作業単位（ワークスペース）の概念を持たなかった（`app.py` は 127.0.0.1 バインド＋ブラウザ自動起動）。既存の「ログイン／認証／セッション」は**クロール対象サイトへのサイト認証**（ADR-0002）を指し、利用者本人の識別とは無関係である。

社内サーバ／別環境への展開（`WEBSPEC2DOC_TRUSTED_HOSTS`）を見据え、利用者の識別と作業ワークスペースの選択を追加したい。一方で、現行のローカル無認証運用・自動ブラウザ起動・既存の大規模テスト資産（E2E／quality_harness）を壊してはならない。

用語衝突（「ログイン／認証」が二重の意味を持つ）を避ける必要もあった（→ CONTEXT.md「サイト認証」節と「利用者・ワークスペース」節で区別）。

## 決定

**メール自己申告による利用者ログインと、ワークスペース選択を、既定 OFF のフラグ機能として導入する。**

1. **認証モデル = 自己申告のメール識別**。パスワードも到達確認も行わず、入力メール＝本人としてセッションを張る。暗号的な認証ではなく「利用者の識別と作業単位の選択」と位置づける。UI・ドキュメントで「セキュアなログイン」とは称さない。
2. **既定 OFF**。`WEBSPEC2DOC_AUTH` 有効時のみガードが働く。未設定なら現行どおり完全無認証で、既存テスト・ローカル UX を維持する。
3. **既存サイト認証と分離**。利用者認証は新規 `auth` ブループリント（`/auth/*`）に置き、既存 `/api/login/*`（サイト認証）と名前空間を分ける。
4. **ワークスペースのデータ物理分離は行わない（段階導入）**。選択したワークスペースはセッションに保持して表示するが、成果物・履歴・`viewpoints.db` のテナント単位分割は本フェーズの対象外とする。
5. **保存**。`instance/auth/`（gitignore 対象）に users / tenants / memberships を JSON で永続化。初回ログイン時に既定ワークスペース `My Workspace` を自動作成し owner を付与。

## 棄却した代替案

- **本物の認証（パスワード＋ハッシュ、またはマジックリンク）**：メール送信基盤や資格情報管理が必要でスコープ過大。「メールのみで良い」という要件に照らし過剰。
- **常時 ON（フラグなし）**：既存 E2E・自動ブラウザ起動・localhost 単一利用者運用を全面的に壊す。回帰リスクが見合わない。
- **ワークスペース単位の完全データ分離を一括実装**：全 route／service に及ぶ大規模改修で回帰リスク大。まず選択のみ導入し、分離は別 ADR で扱う。

## 帰結

- `web/auth/`（models / store / session / guard）＋ `web/routes/auth.py` を新設。`web/__init__.py` で secret_key 設定とフラグ連動ガード登録。
- 用語衝突を CONTEXT.md で解消：「認証」は対象サイト側のみに用い、利用者側は「利用者ログイン」、作業単位は「ワークスペース」（コード識別子は tenant）とする。
- **データ分離は未実装のまま**。マルチ利用者で成果物を共有する状態であり、分離が必要になった時点で後続 ADR を要する（この ADR はその判断を先送りしたことを明示する）。先送り事項の分離方式は `docs/design/workspace-data-separation.md` に設計済み（推奨: パス名前空間化）。実装着手時に ADR-0005（仮）を起こす。
- 設計の詳細は `docs/design/auth-tenant-integration.md`。
