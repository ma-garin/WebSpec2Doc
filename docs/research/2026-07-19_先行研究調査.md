# 先行研究調査 — WebSpec2Doc の位置づけと取り込める知見

- 作成日: 2026-07-19
- 調査方法: Web検索 12回（英語文献中心）。**体系的レビューではない**。書誌情報は原典での再確認を要する。CiNii（日本語文献）は未調査。
- 目的: (1) 本システムの学術的な位置づけを明確にする (2) 実装へ取り込める知見を抽出する

## 要約

本システムの中核（実行中Webアプリからのモデル推論→テスト生成）は **20年以上の研究蓄積がある領域**で、2025〜2026年にはLLMの波が到達し、ほぼ同一アーキテクチャの論文も出ている。一方で、(a) 文書要件と実測の突合を第一級入力とする構成、(b) evidence-only の主張境界を成果物設計に埋め込む方針、(c) 検収証跡への整形は、先行との差分として立てられる。

取り込むべき知見の筆頭は **①メタモルフィックテスト（オラクル無しで使える検証則）②SSIMによる画像比較（画素比較の既知の弱点の解消）③フレイキー分類の学術的裏付け（async-waitが最大要因）④状態抽象化（閾値ベースの既知の限界）** の4つ。

## 1. 領域別の知見と本システムへの示唆

### 1.1 動的モデル推論（本システムの土台）

| 文献 | 要点 |
|---|---|
| Mesbah et al., **Crawljax**（ACM TWEB 2012） | AJAX/SPAをイベント駆動でクロールし state-flow graph を推論。状態同一性はDOM編集距離＋閾値 |
| Memon, **GUI Ripping**（WCRE 2003）/ GUITAR | 実行中GUIからevent-flow graphを抽出しテスト生成。発想の源流 |
| Utting, Pretschner, Legeard（STVR 2012, 22:297-312） | MBT分類の基礎論文。頂点/エッジ網羅は古典的なグラフ網羅基準 |

**示唆**: 本システムの画面遷移図・網羅基準は確立された手法の実装であり、ここ自体に新規性は無い。差別化は入力（文書）と出力（証跡）の側にある。

### 1.2 状態抽象化・近似重複検知 ⭐実装提案あり

| 文献 | 要点 |
|---|---|
| Yandrapally et al., **Near-Duplicate Detection in Web App Model Inference**（ICSE 2020） | 近似重複状態の検知は未解決問題。**閾値の選定自体が本質的困難** |
| **FRAGGEN** | 閾値を使わず**ページ断片化に基づく状態等価**で判定 |
| Tree Kernel によるDOM類似（arXiv 2108.13322） | 木構造カーネルでDOM類似を判定 |
| 探索戦略と状態抽象の実証研究（arXiv 2606.16650, 2026） | 抽象化関数 SAF=(距離関数, 閾値) の選択がモデル品質を左右 |

**示唆**: 本システムの状態同一性判定（URL・DOM比較）は、SPAで状態爆発／過剰併合を起こす既知の弱点を持つ。**閾値チューニングで戦うのではなく、FRAGGEN型のフラグメントベース等価へ寄せるのが研究の方向性**。

### 1.3 文書（要件）からのテスト生成 — 文書駆動MBTの直接先行

| 文献 | 要点 |
|---|---|
| **AI-Driven Test Case Generation from NL Requirements: Survey**（arXiv 2606.06563, 2026） | RBTG（要件ベーステスト生成）の包括サーベイ。LLM化で**幻覚・トレーサビリティ低下・評価の不統一**が新リスクと明記 |
| NLP-Based Requirements Formalization（CEUR 2951） | ルールベース抽出＋NERで要件を構造化 |
| Boilerplate approach（JSS 2024） | **統制された記法（boilerplate）で書かれた要件**からの抽出は精度が大きく上がる |

**示唆**:
- 本システムの「ルールベース抽出・未突合要件は未突合のまま提示」は、サーベイが指摘するLLM経路のリスク（幻覚・トレーサビリティ低下）への設計的回答になっており、**位置づけとして主張できる**。
- 実務改善: **要件記述テンプレート（boilerplate）をユーザーに提供**すると抽出精度が構造的に上がる。「〈画面〉で〈操作〉すると〈結果〉」形式の記法ガイドを文書駆動モードのUIに添える。

### 1.4 オラクル問題とメタモルフィックテスト ⭐実装提案あり（最有力）

| 文献 | 要点 |
|---|---|
| Chen et al., **Metamorphic Testing**（ACM Computing Surveys 51(1), 2018） | オラクル不在でも**複数実行間の関係（MR）**で検証できる |
| Metamorphic Testing of RESTful Web APIs（TSE 2017） | Web API応答の正しさをMRで自動判定 |
| Metamorphic Security Testing for Web Systems（ICST 2020） | セキュリティ性質もMRで表現可能 |

**示唆**: 本システムは evidence-only 原則により「品質判定をしない」が、**MRは観測事実の間の整合性チェックなので主張境界を破らない**。LLM不要・決定的で、本環境の制約にも合う。導入候補のMR:
- 絞り込み検索の結果 ⊆ 絞り込み無しの結果
- ページネーション全ページの合計件数 = 表示件数
- 同一データはビューポートを変えても同一（**実装済みのマルチビューポート差分は実質MRであり、この語彙で説明できる**）
- ソート順を変えても要素集合は不変

### 1.5 ロケータの頑健性・テスト修復 ⭐実装提案あり

| 文献 | 要点 |
|---|---|
| Leotta et al., **Multi-Locators**（2015） | 複数属性の重み付き多数決でロケータを頑健化 |
| **Similo / VON Similo** | 多属性類似度で新版の同一要素を特定。精度 83.8% → 94.7% |
| **VISTA**（視覚ベース修復）/ WATER・WATERFALL | 画像・差分ベースのテスト修復 |
| 業界知見 | 自動修復の最大リスクは**マスキング**（直った風のテストが真の破壊を隠す） |

**示唆**:
- クロール時に既に `element_id` / `name` / `aria_label` / `role` / `placeholder` を実測している。**Similo型の多属性ロケータ候補列（ランク付き）を spec.ts 生成時に埋め込む**と、UI変更時の test_rot を減らせる。
- ただし研究が警告するマスキングを踏まえ、**自動で張り替えず「修復候補の提示」に留める**のが本システムの主張境界と整合する。

### 1.6 フレイキーテスト分類 ⭐実装提案あり

| 文献 | 要点 |
|---|---|
| Luo et al.（FSE 2014） | フレイキー根本原因10分類。**最多は Async Wait**、次いで並行性・実行順依存 |
| Google（Micco, 2016） | 自社テストの**約16%が何らかのフレイキー性**を示す |
| UIフレイキーの実証研究（arXiv 2103.02669） | UIテスト特有のフレイキー要因を分析 |

**示唆**: `failure_classifier` の分類（env_issue / test_rot / app_change）に、研究上最大の要因である **async_wait（タイミング起因）を第4分類として追加**する価値が高い。エラーメッセージの `Timeout` `wait` パターンで機械判定できる。さらにGoogle実務の**同一テスト即時再実行で「決定的失敗かフレイキーか」を切り分ける**手法は、AutoRunの失敗後処理に組み込める。

### 1.7 ビジュアルリグレッション ⭐実装提案あり

| 知見 | 要点 |
|---|---|
| 画素比較の既知の弱点 | アンチエイリアス・フォントレンダリング差・カーソル点滅で**恒常的に偽陽性** |
| **SSIM**（構造的類似度） | 人間の知覚と相関が高く、レンダリング環境差に頑健。実務ツールの主流 |
| pHash | 寛容すぎて実回帰を見逃す（本システムの用途には不適） |

**示唆**: `screenshot_diff.py` は画素差分率ベースで、研究・実務の双方で既知の偽陽性問題を抱える。**SSIM併記への移行**が妥当（Pillow/numpy は導入済みで追加依存なし）。誤検知フィルタ（diff_ignore）と同じ思想で「除外ではなく判定方式の改善」ができる。

### 1.8 トレーサビリティリンク回復

| 文献 | 要点 |
|---|---|
| RAGによるトレーサビリティ回復（REFSQ 2025） | 検索拡張生成で要件間リンクを回復 |
| **TraceLLM**（Requirements Engineering誌, 2026） | プロンプト工学でトレーサビリティを強化 |
| 誤りパターンの知見 | LLMは**「幻のリンク（phantom links）」**を生む。レビューUIと人の確認が必須と結論 |
| テキスト類似だけでは不足（arXiv 2509.05585） | 意味構造の考慮が必要 |

**示唆**: 本システムの「突合できない要件を突合できないまま出す」方針は、この分野の結論（phantom links対策に人の確認が必須）と一致しており、**設計判断の学術的裏付けとして引用できる**。将来LLM補助を入れる場合も「候補提示＋人の確定」の2段構えが研究の指す形。

### 1.9 API仕様の逆生成

| 文献 | 要点 |
|---|---|
| **D2Spec**（MSR 2018） | 文書ページからAPI仕様を抽出（precision 80%台） |
| **APISENSOR**（arXiv 2603.23852） | **実行時トラフィックから**API仕様を発見（本システムのP2-1と同方向） |
| gDoc（2023） | パラメータ間関係のマイニングで構造化文書生成 |

**示唆**: P2-1（観測呼び出しからのOpenAPI雛形）は APISENSOR と同じ「文書よりランタイムが真実」路線で、方向性は研究と一致。APISENSOR はパス正規化・スキーマ推定でより進んでおり、拡張時の参照先になる。

### 1.10 LLMエージェントによるWebテスト（2025-2026の現在地）

| 文献 | 要点 |
|---|---|
| **arXiv 2506.02529**（2025-06 初出, 2026-02 改訂） | **画面遷移グラフ＋LLMでE2Eテスト生成。フォームは状態グラフで扱う。本システムに最も近いアーキテクチャ** |
| **WebTestPilot**（arXiv 2602.11724） | 自然言語仕様からオラクルを推論してE2E実行。完走率99%を報告 |
| **WebTestBench**（arXiv 2603.25226） | E2E自動テストエージェントの評価ベンチマーク |
| AI Agents for Web Testing: Case Study in the Wild（arXiv 2509.05197） | 実サイトでのエージェント適用の実証 |

**示唆**:
- 2506.02529 の存在は「画面遷移グラフ＋テスト生成」がもはや単独では新規でないことを意味する。**本システムの立ち位置は「LLM無しで決定的に動き、主張境界と証跡を持つ」対極設計**として明確化できる。
- WebTestPilot の「自然言語仕様からオラクル推論」は文書駆動モードの発展形に相当する。ただし幻覚リスクは1.3のサーベイが指摘する通りで、本システムがそこを踏まない設計であることが差別化になる。
- **WebTestBench で自システムを評価する**ことは、実力の客観測定として検討に値する。

## 3. 研究計画書向け: 位置づけの整理

**わかっていること（先行あり）**: 実行中Webアプリからの状態遷移モデル推論（Crawljax系）／モデルからのテスト生成（GUITAR系・MBT）／LLMによるE2E生成（2506.02529, WebTestPilot）。

**本システムが立てられる差分**:
1. **文書要件×実測の突合を第一級入力**とし、突合不能を隠さない（phantom links 対策の研究的結論と整合）
2. **evidence-only の主張境界**を成果物スキーマ（claim_scope）として実装に埋め込む — オラクル問題（Barr et al., TSE 2015）への「判定しない」という応答
3. **検収・監査証跡への整形**（実行記録・改竄検知）を出口に置く — 研究ツールが扱わない実務要件

**未検証の弱み（正直に書くべき点)**: 実案件での有効性評価が無い。状態抽象化は閾値ベースで既知の限界を持つ。画像比較は画素ベースで偽陽性が既知。
