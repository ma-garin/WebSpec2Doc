# 機能拡張ロードマップ — 現新比較検証・UX 検証との繋ぎ込み

ステータス: 提案（2026-07-03）。市場調査に基づく開発ロードマップ。事業計画は docs/09、社内ベンチャー準備は docs/10 を参照。

## 0. 出発点

商用テスト検証サービス（ベリサーブ等）のサービスラインである
**「現新比較検証サービス」**（OS 移行・ブラウザ VerUP 時の現行/新システム比較。移行時テストの最大 80% を占めるとされる）と
**「UX・ユーザビリティ検証サービス」**（エキスパートレビュー・ユーザテストによる利用価値向上）を、
既存の開発テーマ 3 件（iframe/Shadow DOM 対応・LLM 観点の要素レベル evidence・残バックログ）と繋ぎ込み、WebSpec2Doc を一段レベルアップさせる。

### 決定的な接続点

現新比較検証の代表的な課題は次の 4 つとされる:

1. 比較の正確性
2. 検証コストの抑止
3. 短納期
4. **設計ドキュメントの不備**

このうち 4 は、WebSpec2Doc の中核機能「URL から仕様書を復元する」がそのまま解決策になる。
画像比較ツール（T-DASH・MagicPod・ATgo 等）は 1〜3 しか扱えない。**「仕様の比較」と「ドキュメント復元」を同時にできる現新比較ツールは調査した範囲で存在しない**。ここが本システムの参入角度になる。

## 1. 市場調査サマリ

### 1-1. 現新比較・マイグレーション市場（追い風が強い）

- 国内 IT モダナイゼーションサービス市場は 2025 年で約 1 兆 3,044 億円（前年比 +10.1%）、CAGR 10.2% で 2030 年に約 2 兆 1,234 億円と予測（IDC）。大企業・中堅企業の約 8 割がレガシーシステムを保有
- 「2025 年の崖」（経産省）: レガシー依存を続けた場合、年間最大 12 兆円の経済損失リスク
- 移行時のテストは工数の最大 80% を占めるとされ、現新比較の自動化ツールが複数商用化されている（T-DASH の画像比較、MagicPod の画像差分、ATgo の AI 画像比較（2024 年に画像内テキスト読取り差分を追加））
- 既存ツールは**ピクセル/画像比較が主軸**。仕様（フォーム構造・バリデーション・画面遷移・カバレッジ）の比較まで踏み込むものはない

### 1-2. ビジュアルリグレッションテスト（技術の参照点）

- Percy / Chromatic / Applitools が 3 強。Applitools は Visual AI による「意味のある差分」抽出、Percy は 2025 年末に AI Visual Review Agent（誤検知の約 4 割を自動フィルタ、レビュー時間 1/3）を投入
- 教訓: **ピクセル差分は誤検知との戦い**。レンダリングゆらぎの吸収（アンチエイリアス・動的コンテンツのマスク）と「意味のある差分」への絞り込みが製品価値の中心。既存の `src/diff/screenshot_diff.py` を現新比較に転用する際は、この誤検知抑制の設計が最重要

### 1-3. AI テストエージェント市場

- AI テストツール市場は 2025 年 6.9 億ドル → 2035 年 38 億ドル予測
- 「AI 支援型」（mabl・Autify・Katalon: 人のワークフローを AI が補助）と「自律型」（testRigor・QA Wolf 等: 意図や観測から AI がカバレッジを自律生成）に二極化
- WebSpec2Doc は「自律探索でテスト設計の**入力（仕様）**を作る」ポジションで、両陣営の上流に位置する（docs/09 の差別化と整合）

### 1-4. UX・アクセシビリティ自動評価

- 自動 a11y 検査の定番は axe-core（MIT ライセンス、DOM があれば動作、WCAG 基準に紐づく構造化違反を返す）。Lighthouse の a11y カテゴリも内部は axe-core
- ただし自動検査で捕捉できるのは a11y 問題全体の 30〜40% → 「ツールによる一次判定＋人の二次判定」がベリサーブ含む業界標準の構え
- LLM によるヒューリスティック評価（ニールセン 10 原則）は研究が進むが限界も明確:
  - GPT-4o の評価は人間専門家の指摘との一致率 21.2% にとどまり、幻覚による誤検知が発生（2025 年比較研究）
  - 30 サイト・850 件超の評価で検出一致率 84%・Cohen's Kappa 0.50（中程度）という報告もあり「早期スクリーニング用途なら実用域」
  - → **evidence-only 原則（セレクタ・bbox・スクリーンショットに紐づかない指摘は出さない）を LLM UX 評価に適用すれば、幻覚問題への直接の対策になる**。これは本システムが既に持つ設計思想の自然な延長
- コードからの仕様復元ツール（EPAM ART、Reversa 等）は登場しているが、いずれも**ソースコードが必要**。URL だけで動く本システムの差別化は維持されている

## 2. 繋ぎ込みの全体像

```text
                    ┌──────────────────────────┐
                    │ Sprint A: iframe/Shadow DOM 対応│ ← 両サービスの土台
                    └────────────┬─────────────┘
              ┌──────────────────┴──────────────────┐
   ┌──────────▼──────────┐              ┌───────────▼───────────┐
   │ Sprint B: 現新比較モード │              │ Sprint C: UX 自動レビュー │
   │ （現新比較検証サービス対応）│              │ （UX検証サービス対応）      │
   └──────────┬──────────┘              └───────────┬───────────┘
              └──────────────────┬──────────────────┘
                    ┌────────────▼─────────────┐
                    │ Sprint D: レポート/ROI 統合    │
                    └──────────────────────────┘
```

- **iframe/Shadow DOM（テーマ1）** は両サービスの前提。現新比較の主戦場であるレガシー業務システムは frameset/iframe 構成が多く、モダン側は Web Components（Shadow DOM）が増えている。どちらも今のクローラは素通りする
- **LLM 観点の要素レベル evidence（テーマ2）** は UX 自動レビューの実装機構そのもの。「どの要素を根拠にした指摘か」を強制することが幻覚対策になる
- **残バックログ（テーマ3）** のうち MutationBlocker 遮断の監査ログ化・カバレッジギャップ提示は、現新比較の「比較の網羅性証明」（どこまで見た/見ていないの明示）として昇格する

## 3. スプリント計画

### Sprint A: iframe/Shadow DOM 対応（土台・約1週間）

- Playwright は open な Shadow DOM を CSS ロケータで自動貫通するため、要素抽出（フォーム・リンク・アクション）の収集 JS を `shadowRoot` 再帰走査に拡張するのが主作業。closed shadow root は「検出したが読めない」として evidence に記録（未確認を未確認と言う）
- iframe は `frame_locator` / `page.frames` で走査し、フレーム内のフォーム・リンク・見出しを親ページの仕様に統合。クロスオリジン iframe は「外部埋め込み」として記録のみ
- 受け入れ条件: デモサイトに iframe ページと Web Components ページを追加し、実ブラウザ E2E でフォーム抽出・状態検出が通ること

### Sprint B: 現新比較モード（約2週間）

ベリサーブの検証フロー（対象システム調査 → 対象画面・帳票整理 → テストケース作成 → テスト実行 → 結果確認）のうち、下線部を自動化する:

1. **2 ターゲットクロール**: 現行 URL と新 URL のペアを受け取り、それぞれをクロール（既存 crawl_urls を 2 回実行＋対応付け）
2. **画面対応付け**: URL パス・タイトル・fingerprint の類似度で現新の画面ペアを自動マッチング（未対応画面は「追加/削除」として報告）
3. **三層比較**:
   - 仕様差分（既存 diff 層を転用: フォーム構造・バリデーション・遷移・画面インベントリ）
   - 画像差分（既存 screenshot_diff を転用。動的領域マスクと閾値設定で誤検知抑制）
   - リンク切れ・HTTP エラー検査（チラシの想定不具合「リンク切れによる操作不可」に対応）
4. **不具合分類レポート**: 表示崩れ / 文字化け・意味消失 / 理解不可能 / 操作不可 の 4 分類（チラシの想定不具合分類に準拠）で報告。全指摘に evidence（現新両方のスクリーンショット・セレクタ）を付与
5. 出力: report.html に「現新比較」タブ、Excel に比較結果シート

差別化: 画像比較ツールと違い、**復元済み仕様書が同時に手に入る**（課題「設計ドキュメントの不備」の解消）。比較は仕様レベルなので「ピクセルは違うが仕様は同じ」（フォント差等）と「ピクセルは同じだが仕様が壊れた」（required 属性消失等）を区別できる。

### Sprint C: UX 自動エキスパートレビュー（約2週間）

「ツール一次判定＋人の二次判定」の一次判定部分を提供する:

1. **a11y 検査（rules 層・confidence 1.0）**: クロール時に axe-core（MIT）を注入実行し、WCAG 違反を画面ごとに evidence（セレクタ・違反ルール ID）付きで記録
2. **ヒューリスティック評価（LLM 層・confidence ≤ 0.9）**: ニールセン 10 原則を観点テンプレート化し、画面のスクリーンショット＋抽出済み要素インベントリを入力に指摘を生成。**要素レベル evidence を必須にし、実在しないセレクタを参照する指摘は破棄**（幻覚フィルタ）。LLM なしでは rules ベースのチェック（ラベル無し入力・コントラスト・タップ領域サイズ等）にフォールバック
3. 出力: report.html に「UX 所見」タブ（重大度 × 画面のマトリクス）、確信度と根拠を明示し「自動検査は a11y 問題の 30〜40% を捕捉する一次スクリーニングである」旨を免責表示（evidence-only 原則）

### Sprint D: レポート/ROI 統合（約1週間）

- usage_tracker に現新比較・UX レビューの実績を追加（比較画面数・検出差分数・UX 指摘数 → 削減工数の係数化）
- 残バックログ消化: MutationBlocker 遮断の監査ログ化、カバレッジギャップ（未探索領域）の明示をレポートに追加 → 現新比較の網羅性証明として表示
- デモ台本に「移行検証デモ」（DemoMart の新旧 2 バージョンを比較）と「UX 診断デモ」を追加

## 4. 事業面の含意（詳細は docs/09・10 — 開発とは別軸）

- サービスメニューに「**移行検証支援**」（現新比較 + 復元仕様書の納品）と「**UX 簡易診断**」（一次スクリーニングレポート）を追加できる。前者はモダナイゼーション市場（1.3 兆円・年 10% 成長）に直結し、docs/10 の「半年 1 億」ミックスの単価根拠を補強する
- ベリサーブ型の人的検証サービスと競合ではなく、**その一次判定ツールとして補完**するポジションも取れる（協業余地）
