## クレジット節約方針（明示依頼がない限り順守）

- まず最小限の調査で原因・方針だけを簡潔に報告する。
- 大きなファイル全文取得・広範囲検索・大きな API レスポンス取得を避ける。
- 2分以上かかりそうな調査・作業は、中断してユーザーに確認する。
- 修正・コミット・push・デプロイまで進める場合は、ユーザーの明示的な依頼または承認を得る。

## GitHub 証跡の恒久ルール

- 本リポジトリではユーザー承認済みの恒久ルールとして、実装・修正の完了時は個別の再承認を待たず、コミット、topic branch へのpush、PR作成、CI確認、mainへのsquash merge、ローカルmain同期まで行う。
- 長時間作業の中断・セッション終了が近い場合は、追跡可能なcheckpoint commitを作成し、topic branchへpushする。未完成の変更はmainへマージしない。
- 無関係な差分はPRに含めず、CIが失敗した状態ではマージしない。

## UI変更時の必須フロー（機械的ゲート付き）— INC-2026-001 是正済み

UI（HTML / JS / CSS）を変更する場合は、必ず次の順で進める。
**省略は INC-2026-001 相当のインシデントとして扱う。**

1. **スキャン**: 変更対象・影響範囲・既存UI構成を把握する。
2. **提示**: 実装方針・期待される画面状態をユーザーへ提示する。
3. **合意**: ユーザーの合意を得てから実装する。
4. **L1/L2 テスト**: `make test` → 全 PASS を確認する。
5. **L3 E2E テスト（機械的強制・省略禁止）**: `make verify-ui` を実行する。
   - E2E テストが全 PASS → `.ui-verified` マーカーが自動生成される
   - FAIL → コードを修正してから再実行する（コミット不可）
6. **スクリーンショット確認**: `tests/e2e/screenshots/` の画像を目視確認する。
7. **解像度確認**: 1920×1080 / 1366×768 の両方でブラウザを確認する。
8. **フロー確認**: 変更した機能のユーザーフローを最初から最後まで通して操作する。
9. **コミット**: `git commit` → pre-commit hook が自動的にゲートを確認する。
10. **最終報告**: E2E 結果・スクリーンショット・目視確認内容を明記して報告する。

### ゲートの強制構造（人間の判断に依存しない）

```
make verify-ui
  → Flask 起動 → E2E テスト実行 → 全 PASS → .ui-verified 生成
                                   ↓ FAIL
                                コミット不可（修正必須）
git commit
  → pre-commit hook
       → .ui-verified が存在 かつ 1時間以内 → PASS → コミット完了
       → 不在 or 古い                        → BLOCKED
```
