# WebSpec2Doc

稼働中の Web システムの URL から、QA テスト分析インプット文書を自動生成するツール。一度登録したサイトを再クロールして仕様ドリフトを検知する継続利用を想定する。

## Language

### UI フロー

**ウィザード**:
サイトの新規登録・クロール実行をガイドする4ステップのUI。各ステップは独立したページとして表示され、前後のボタンで移動する。

**解析ステップ（ステップ1）**:
URL入力とページ解析を行うウィザードの第1ページ。解析完了後はサマリー（検出件数・ログイン必要件数）のみを表示し、「次へ」ボタンで条件設定ステップへ進む。

**条件設定ステップ（ステップ2）**:
クロール対象の画面選択・ログイン（ID/PASSWORD）・差分出力オプションを設定するウィザードの第2ページ。再クロール時は前回の設定を復元した状態で開く。

**実行ステップ（ステップ3）**:
オートクローリングの進行ログを表示するウィザードの第3ページ。完了後に「レポートを見る」ボタンが出現する。

**画面遷移図**:
クロール結果からサイトの画面間遷移を可視化したビュー。シーケンス図（操作順）/コミュニケーション図（全体関係）/アクティビティ図（テスト手順）のUML 3タブに加え、テスト観点マップで遷移をQA観点へ分類する。各図のエッジ番号と対応表のNoを揃える。全ページの50%以上から出るリンクは共通ナビゲーション（ヘッダー/フッター）と判定して非表示にし、意味のある遷移のみを表示する。フォーム送信は「フォーム送信」イベントで区別する。
_Avoid_: 図だけで遷移詳細を詰め込む。図と表を分断して番号対応を失う。

**画面遷移表**:
ISTQBの状態遷移テスト技法に基づく標準テーブル。行に現在の画面、列にイベント/条件・遷移先画面・アクションを持つ。QAテスターがテストケース設計の起点として使う。クローラーが収集したリンクとフォーム情報から機械的に生成する（イベント種別はリンク/フォーム送信を判別）。
_Avoid_: 遷移一覧・リンク表

**レポートステップ（ステップ4）**:
生成ドキュメントを確認するウィザードの第4ページ。タブ構成：概要 / 画面別仕様 / テスト条件 / 設計 / 技法詳細 / 画面遷移図 / 遷移表 / 履歴・差分（計8タブ）。エクスポートはヘッダーのドロップダウンから行う。

### サイト・クロール

**サイト**:
監視対象として登録された 1 つの Web システム。ドメインで識別され、クロール設定（対象URL・方式・深さ・出力形式・認証）を保持する永続レコード。
_Avoid_: ドメイン（出力フォルダのキーとしては domain を使うが、ユーザー概念は「サイト」）

**オートクローリング**:
入力 URL を起点にリンクをたどって到達ページを自動収集するクロール方式。本ツールの基本動作（メイン）。
_Avoid_: スクレイピング、巡回

**ページ解析**:
オートクローリング実行の前段で、到達ページの一覧（URL・タイトル）を先に取得する軽量工程。対象範囲とログイン必要箇所を見極めるために使う。認証後にもう一度走らせて認証後ページを一覧化する（再解析）。
_Avoid_: discover（コード内識別子としては discover を使う）

**再クロール**:
登録済みサイトを保存済み設定で取り直すこと。前回スナップショットとの差分（仕様ドリフト）を検知する継続利用の中核。

### サイト認証（クロール対象サイトへのログイン）

この節の「ログイン／認証／セッション」は、すべて**ツールがクロール対象サイトへログインする**ことを指す。
アプリ利用者本人のログインは別概念（→ `### 利用者・ワークスペース` の「利用者ログイン」）。混同しないこと。

**ログイン必要箇所**:
ページ解析の途中で、認証によりアクセスがブロックされる地点（ログインページへのリダイレクト・パスワードフォーム・401/403）。ツールはこれを検出してユーザーに提示する。

**自動ログイン**:
ページ解析で検出したログインページのフォームフィールドをGUI上に動的表示し、ユーザーがID・パスワード等を入力してツールが自動送信する認証取得方式。送信後に `detect_login_wall()` で成功を判定し、MFA等の追加フィールドが出現した場合はGUIに動的追加して続行する。入力されたID・パスワードはセッション確立後に即破棄し、保存しない。
_Avoid_: 手渡しログイン（別ウィンドウ起動方式は廃止）

**セッション**:
自動ログインで取得し保存する認証状態（storage_state＝Cookie 等）。サイト単位で `output/{domain}/auth.json` に保存し再クロールで再利用する。期限切れはクロール中に検出し再ログインを促す。セッションにはID・パスワード本体を含まない。
_Avoid_: 認証情報（credentials はパスワード本体を連想させるため避ける）

### 利用者・ワークスペース

アプリ利用者の識別と、作業単位の切り替えに関する概念。上の「サイト認証」（対象サイトへのログイン）とは完全に別物。
商用/共有サーバ向け機能で、`WEBSPEC2DOC_AUTH_MODE`（既定 `auto`）で制御する。実装: `web/auth.py`＋`web/services/auth_store.py`＋`web/tenancy.py`＋`web/routes/account.py`（詳細 `docs/AUTH_TENANCY.md`）。

**利用者ログイン**:
アプリ利用者本人が WebSpec2Doc に入るための認証。商用/共有サーバ向けに、パスワード・ロックアウト・アカウント管理・API トークンを備える（実装 `web/auth.py` ほか、`docs/AUTH_TENANCY.md`）。セッションに利用者とテナントを保持する。
_Avoid_: 認証（「認証」は対象サイトへのサイト認証を指すため、利用者側には使わない）／サインイン

**利用者**:
メールアドレスで識別される WebSpec2Doc の使い手。1 人が複数のワークスペースに所属しうる。
_Avoid_: アカウント（User と Customer の混同を避け、素直に「利用者」）

**ワークスペース**:
成果物・設定・観点DBをテナント単位で分離する作業単位。初回は `/auth/setup` で最初のワークスペースとオーナーを作成する。有効時、成果物は `output/tenants/{slug}/`、観点DBは `instance/tenants/{slug}/viewpoints.db` に分離される。
_Avoid_: テナント（ユーザー概念は「ワークスペース」。コード識別子としては tenant / tenant_id を使う ← 「サイト／domain」と同じ前例）

**メンバーシップ**:
利用者とワークスペースの所属関係。ロール（owner / admin / member）を持つ。どのワークスペースを選べるか、選択が正当かの判定根拠になる。

### QA生成・自動テスト

**QAプロセス**:
クロール結果（report.json）を入力に、テスト計画・テスト分析・テスト設計・テストケース・横断レビューの5文書を生成するビュー。OpenAI API（OPENAI_API_KEY）が設定されている場合は補完・品質向上を行い、未設定の場合はローカルテンプレートでフォールバック生成する。実装: `web/routes/qa_process.py`（薄いルート）＋`web/services/qa/`（doc_generator/advanced_generator/advanced_html/helpers）。
_Avoid_: QAドキュメント生成、テスト文書自動作成

**高度QA成果物**:
QAプロセスの拡張機能として生成する3種の追加成果物。①モデルグラフ（画面間遷移と状態をHTML可視化）②Playwright候補（自動テスト対象フォーム・ボタンのJSON/HTML）③品質観点（ISTQBに基づく観点カード）。`/api/qa-process/generate-advanced` で一括生成。

**AutoRun**:
URLを入力するだけでページ解析→クロール→QA生成→Playwright spec.ts 生成→テスト実行を全自動で行うビュー。ジョブ単位で状態管理（idle/discovering/crawling/generating_qa/generating_scripts/running_tests/complete）し、ポーリング方式でフロントに進捗を配信する。実装: `web/routes/auto_run.py`＋`web/services/spec_ts_generator.py`＋`web/services/playwright_executor.py`。
_Avoid_: 全自動テスト、ワンクリックテスト

**実行方針**:
AutoRun でテストを実行する前にユーザーが選択するテスト実行の方針。全件実行・スモークのみ・重要画面のみ等の粒度を指定できる。承認ステップ（awaiting_approval）でユーザーが確認してから実行に進む。

**観点管理**:
テスト観点をツリー分類＋インラインテーブルの3ペインUIで管理するビュー。手動追加・バルク操作・AI提案（ルールベース/LLM）に対応する。実装: `web/routes/viewpoints.py`＋`web/services/viewpoint_store.py`。
_Avoid_: 観点リスト、チェックリスト

### 品質・根拠

**evidence（根拠）**:
生成された仕様・テスト条件・観点に紐づく実測の根拠（セレクタ・スクリーンショット座標・confidence）。実在する要素に紐づかない生成物は破棄する（evidence-only 原則）。

**クロール礼儀**:
対象サイトへ負荷や副作用を与えないための既定動作。robots.txt 尊重・per-origin レート制御・破壊的リクエスト（POST/DELETE等）の遮断からなる。実装: `src/crawler/politeness.py`＋`network_interceptor.py`。

**ROIダッシュボード**:
利用実績（クロール数・生成文書数）から削減工数（時間・円）を明示係数で推定表示するビュー（`/usage`）。係数は環境変数で調整可能で、推定である旨を常に明記する。
