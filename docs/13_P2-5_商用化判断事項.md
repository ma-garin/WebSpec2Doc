# P2-5 その他（商用化判断事項）の到達点と残件

- 作成日: 2026-07-19
- 対象: `docs/12_実務化施策ロードマップ.md` P2-5
- 位置づけ: 実装で完了させた項目と、**実装では完了にできない項目**を分けて記録する

## 要約

P2-5 は「商用化判断後」と条件付きで括られた雑多な項目群だった。棚卸しした結果、
**技術的に実装できるもの**と、**事業判断・外部監査が前提で実装では終わらないもの**に
分かれた。前者は実装・マージ済み、後者は着手条件を明示した状態で保留とする。

「全部やった」と言えない項目を、言えないまま記録しておくことが本書の目的である。

## 実装して完了させた項目

| 項目 | 実装 | 備考 |
|---|---|---|
| 完全アーカイブ（規制業種向け） | `src/archive/full_archive.py` | 保持ポリシー（P0-4）の上に構築。マニフェスト（SHA-256）同梱で改竄検知可 |
| sitemap 監視 | `src/archive/external_monitor.py` | 監視対象URLの増減を検知。XXE対策に defusedxml を使用 |
| PDF 文書監視 | 同上 | 内容ハッシュで差し替えを検知 |

いずれも主張境界を明示している（`archived_outputs_only` / `fetched_content_changes_only`）。
アーカイブは**元データを消さない**。消す判断は保持ポリシー側の責務であり、削除経路を
二重に持つと事故になるため。

## 実装では完了にできない項目

### SAML / SCIM

- **状態**: 未着手（保留）
- **理由**: ロードマップ上「大企業商談が現実化した時点」と条件付き。OIDC（P1-10）で
  Entra ID / Google Workspace は既に賄えるため、SAML が要るのは「IdP が SAML しか
  出さない顧客」が実在したときに限られる。実顧客が無い状態で作ると、検証もできない
  コードを抱えることになる。
- **着手条件**: SAML必須の顧客が具体化し、テスト用IdP（検証環境）を用意できること。

### SOC 2 / ISO 27001（エディション設計と認証基準）

- **状態**: 実装対象ではない
- **理由**: これらは**外部監査機関による認証**であり、コードを書いて完了する種類の
  ものではない。開発側で「取得した」と宣言することはできない。
- **本製品が現時点で提供できている監査対応の材料**:
  - 管理操作の監査ログ（P0-5）
  - データ保持ポリシーとバックアップ手順（P0-4）
  - 完全アーカイブと改竄検知（本書）
  - テナント分離・認証（既存）・APIトークンのスコープ分離（P1-10）
  - 依存の脆弱性監査（CI の pip-audit）
- **残る事業側アクション**: 認証取得の要否判断、監査機関の選定、統制文書の整備、
  審査対応。いずれも開発作業ではない。

## 判断が要るもの（意思決定の入口）

1. 商用提供の目標時期 — これが決まらないと SAML / 認証取得の優先度が確定しない
2. 想定顧客の規模と業種 — 規制業種が主戦場なら完全アーカイブの運用要件を詰める必要がある
