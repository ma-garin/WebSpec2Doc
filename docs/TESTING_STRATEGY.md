## 1. 目的と適用範囲

本文書は WebSpec2Doc の全ソフトウェアコンポーネントに対する品質保証プロセスを定義する。  
プロセス・ルール・ツールの選定根拠を明示し、チームメンバー全員が同一の品質基準で作業できる状態を維持する。

**適用対象**: `src/`, `web/`, `static/`, `templates/` に含まれる全コンポーネント  
**適用外**: `output/`（生成物）, `venv/`（依存ライブラリ）

---

## 2. テストレベルとゲート基準

ISTQB が定める4テストレベルを全て実施する。任意の段階の省略は禁止。

### L1: コンポーネントテスト（Unit Test）

| 項目 | 内容 |
|------|------|
| **ツール** | pytest >= 8.0 |
| **対象** | 個別関数・クラス・Pure Logic |
| **カバレッジ要件** | `src/`, `web/` の行カバレッジ **80% 以上** |
| **実行コマンド** | `make test` |
| **実行タイミング** | 全コミット前（pre-commit hook により自動） |
| **ゲート** | 1件でも FAIL、またはカバレッジ < 80% → コミット不可 |

### L2: 統合テスト（Integration Test）

| 項目 | 内容 |
|------|------|
| **ツール** | pytest + Flask test client |
| **対象** | API エンドポイント / サービス間結合 / 状態遷移 |
| **カバレッジ要件** | L1 と合算で 80% 以上 |
| **実行コマンド** | `make test`（L1 と同一コマンド） |
| **実行タイミング** | 全コミット前（pre-commit hook により自動） |
| **ゲート** | 1件でも FAIL → コミット不可 |

> **L1+L2 の証明範囲**: 「コードが壊れていない」こと。  
> **L1+L2 が証明しないこと**: 「ユーザーが価値を受け取れる」こと。

### L3: システムテスト（System Test / E2E Test）

| 項目 | 内容 |
|------|------|
| **ツール** | pytest-playwright 0.5.0 / Chromium |
| **対象** | ブラウザ上の実フロー（AutoRun・ウィザード・ダッシュボード・モーダル） |
| **実行コマンド** | `make verify-ui` |
| **実行タイミング** | **HTML / JS / CSS ファイルを変更した全コミット前（必須・強制）** |
| **ゲート** | 1件でも FAIL → `.ui-verified` マーカーなし → コミット不可 |
| **証跡要件** | スクリーンショット自動保存（`tests/e2e/screenshots/`） |
| **解像度** | 1280×800（デフォルト） + 1920×1080（拡張チェック） |

### L4: 受け入れテスト（Acceptance Test / UAT）

| 項目 | 内容 |
|------|------|
| **ツール** | 手動（ISTQB FL レベルのテスト担当者ロールで実施） |
| **対象** | ユーザーストーリー / 非機能要件（使いやすさ・わかりやすさ） |
| **実行タイミング** | 新機能リリース時・大幅 UI 変更時 |
| **ゲート** | ユーザー（プロダクトオーナー）が確認・承認 |
| **証跡要件** | スクリーンショット + コメント記録 |
| **ペルソナ** | ISTQB FL 取得済みジュニアテスト担当者（初見・マニュアルなし） |

## 3. テスト種類マトリクス

| テスト種類 | L1 | L2 | L3 | L4 | 実施条件 |
|-----------|----|----|----|----|---------|
| 機能テスト | ✓ | ✓ | ✓ | ✓ | 全変更 |
| 回帰テスト | ✓ | ✓ | ✓ | — | 全変更 |
| ユーザビリティ | — | — | 一部 | ✓ | UI変更時 |
| セキュリティ | — | ✓ | — | — | 認証・入力処理変更時 |
| 性能テスト | — | — | — | — | 現状対象外（将来拡張） |

## 4. ツールチェーン

| カテゴリ | ツール | バージョン | 用途 |
|---------|--------|----------|------|
| Unit/Integration | pytest | >=8.0 | L1・L2 |
| Coverage | pytest-cov | >=5.0 | カバレッジ計測 |

| Code Review | code-reviewer agent | — | 静的品質確認 |

| Git Gates | pre-commit hook | — | 自動強制実行 |

#### L4 受け入れテスト シナリオ一覧（具体的タスク）

ISTQB FL レベルのジュニアテスト担当者が「マニュアルなし・初見」で実施すること。

| # | シナリオ | 合格基準 |
|---|---------|---------|
| UAT-01 | ダッシュボードを開き、「+ サイトを追加」からサイトを登録してクロールを完了させる | エラーなし・レポートが表示される |
| UAT-02 | AutoRun の URL 入力欄に `https://example.com` を入力して「開始」を押す | 処理が進み進捗ステップが更新される |
| UAT-03 | AutoRun が `awaiting_approval` になったとき、モーダルが自動表示され、「スモークのみ」を選択して承認できる | モーダルが見え・操作でき・テスト実行が開始される |
| UAT-04 | テスト実行完了後に「テスト実行レポートを見る」ボタンを押す | Playwright レポートが表示される |
| UAT-05 | 1366×768 のウィンドウ幅で全機能が操作可能である | 水平スクロールなし・ボタンが切れない |
| UAT-06 | ダッシュボードで登録済みサイトを削除できる | サイトが一覧から消える |
| UAT-07 | 無効な URL（例: `not-a-url`）を入力して「開始」を押す | 分かりやすいエラーメッセージが日本語で表示される |

**実施記録**: シナリオごとにスクリーンショットを撮影し `tests/e2e/screenshots/uat-XX.png` として保存すること。

## 5. 完了基準 (Exit Criteria)

変更がリリース可能な状態の定義は `docs/DEFINITION_OF_DONE.md` に委任する。

テストサイクル完了の定義:
1. L1+L2: 全テスト PASS + カバレッジ 80%+
2. L3（UI変更時）: E2E 全テスト PASS + スクリーンショット確認済み
3. L4（新機能時）: ユーザーによる確認・承認完了
4. CRITICAL/HIGH の未解決指摘事項ゼロ

---

## 6. リスク管理

| リスク | 発生確率 | 影響度 | 対策 |
|--------|---------|--------|------|
| E2E テストがフラキーになる | 中 | 中 | 再試行2回まで許容・原因調査必須 |
| テストメンテナンスの遅延 | 高 | 高 | 機能追加と同一 PR にテストを必須化 |
| L3 省略の習慣化 | 高 | 高 | pre-commit hook + `.ui-verified` で機械的に強制 |
| L4（受け入れ）の省略 | 中 | 高 | DoD にチェックリストを設け記録を残す |

## 7. 本戦略の改訂ルール

- インシデント発生後: 必ず本文書を見直し是正措置を反映する
- 四半期ごと: 有効性をレビューし、陳腐化した箇所を更新する
- 大幅なアーキテクチャ変更時: テストレベルの対象範囲を再定義する
