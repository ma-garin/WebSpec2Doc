背景・課題
現場の実態

当社が業務委託で参画するアジャイル開発・DevOps 現場においては、開発スピードを優先するあまり 仕様書・設計書の整備が後回し になるケースが増加している。検証支援チームが着任した際に「ドキュメントはありません。実際に触って覚えてください」と言われることが珍しくない。

課題の構造
課題影響現状の対処
画面構成・遷移が不明 テスト設計の網羅性が担保できない 手動で全画面を触って一覧を作成（2〜4人日）
入力項目の仕様が不明 境界値・同値分割テストの対象が不明確 開発者にヒアリング（スケジュール調整が必要）
遷移条件・異常系が不明 異常系テストが属人的になる 勘・経験則に依存
ドキュメントの陳腐化 あっても信頼できない 実装を確認して都度アップデート

エグゼクティブサマリー
提案の一言： 「URL を渡すだけで、QA エンジニアがその日からテスト設計を始められる文書一式を自動生成する Python ツールを社内開発する」
ソリューション概要

WebSpec2Doc は、公開 Web サービスの URL を入力として受け取り、Playwright によるブラウザ自動操作でページをクロールし、画面遷移グラフを構築する。そこから QA エンジニアが必要とする以下の文書を自動生成する：

  • 画面一覧表（URL・タイトル・主要要素・遷移先）
  • 画面遷移図（Mermaid フローチャート）
  • 入力項目一覧（フォーム・入力フィールド・推定バリデーション）
  • 画面仕様書 ※ Claude API オプション（目的・操作・テスト観点）
競合・代替手段との差異
手段できることできないこと
Screaming Frog 等 SEO ツール URL 一覧・リンクマップ QA 向け仕様書は出力できない
Slickplan 等サイトマップツール ビジュアルサイトマップ 入力項目・遷移条件は取れない
手動調査 正確・詳細 2〜4人日のコスト / 属人化
WebSpec2Doc（提案） QA 向け文書一式を 5 分で自動生成 ログイン必須領域・完全なビジネスロジック仕様

スコープ・制約条件
スコープ内（対象）
  • ログイン不要の公開 Web ページ
  • クロール深さ 3 以内のページ（設定変更可能）
  • 最大 50 画面 / 実行（設定変更可能）
  • 静的 HTML・SPA（JavaScript レンダリング後）両対応
スコープ外（対象外）
  • ログイン必須ページ（将来的に拡張予定）
  • ビジネスロジック・バックエンドの仕様
  • 非公開 API・内部通信の仕様
  • モバイルアプリ（ネイティブ）
