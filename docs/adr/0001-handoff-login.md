# 手渡しログインをサブプロセス＋シグナルファイルで実装する

## 文脈

認証付きサイト（社内システム・金融/医療/官公庁）をGUIから扱いたい。これらは MFA・SSO・CAPTCHA を伴うことが多く、ID/PW をツールが代入する「自動ログイン」では突破できない。また第三者検証会社はクライアントの資格情報を外部にもツールにも預けられない。

GUIの「手渡しログイン」（CONTEXT.md 参照）では、ローカルのブラウザをユーザーに手動操作させ、確立されたセッション（storage_state）だけを取り込む。問題は「ログイン開始」と「ユーザーがログインし終えるまでの待機」と「ログイン完了の確定」が複数のHTTPリクエストに跨り、その間ブラウザコンテキストを生かしておく必要があること。

## 決定

**自動ID/PW入力は採用せず、手渡しログインのみを提供する。** 実装は **サブプロセス＋シグナルファイル方式** とする。

- `POST /api/login/start`：`src/main.py --login <url>`（headless=False）をサブプロセスとして起動する。Playwright はこのサブプロセス内だけで動く。
- サブプロセスは、所定のシグナルファイル（例: `output/{domain}/.login_done`）が出現するまでポーリングして待機する。
- `POST /api/login/finish`：シグナルファイルを設置する。サブプロセスがそれを検出し `context.storage_state` を `output/{domain}/auth.json` に保存して正常終了する。
- Flask 側はサブプロセスの戻り値で成否を判定し、シグナルファイルを後始末する。

理由:

1. 既存 `/run`・`/api/discover` が同じく `src/main.py` をサブプロセス実行している。一貫した方式で、プロセス分離による堅牢性（クラッシュ隔離・確実な終了）が得られる。
2. Playwright の **sync API はスレッド制約**があり、Flask のリクエストスレッドに sync の Playwright ループを持ち込むと取り回しが複雑になる。サブプロセスに閉じ込めることでこの制約を回避する。
3. リクエスト間状態を「サーバ内メモリのセッションオブジェクト」ではなく「ファイルシグナル」で表現することで、状態が可観測・デバッグ容易になる。

## 棄却した代替案

- **Flaskプロセス内でブラウザ保持**（メモリにトークンで保持し、専用バックグラウンドスレッドが Playwright ループを所有）：sync API のスレッド制約と、Flask 開発サーバのワーカモデルとの相性で実装が複雑。プロセスクラッシュ時にブラウザがリークしやすい。
- **自動ID/PW入力**（フォームにツールが代入し送信）：MFA/SSO/CAPTCHA で破綻し、パスワード本体を扱う必要が生じる。本プロダクトのオンプレ/非保持の思想に反する。

## 帰結

- 既存の `auth.py:capture_auth_state` は端末 `input()` 待ち。これをシグナルファイル待ちに置き換える（CLI の対話モードは別途維持するか要検討）。
- セッションはサイト単位 `output/{domain}/auth.json` に保存（従来のプロジェクト直下単一 `auth.json` から移行）。`output/` は Git 除外済み。
- #7（再クロール時のセッション期限切れ検出）も同じ subprocess 境界を跨ぐため、本ADRの方式を踏襲する。
