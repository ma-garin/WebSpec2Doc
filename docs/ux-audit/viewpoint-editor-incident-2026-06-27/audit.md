# 観点編集UI UX監査（2026-06-27）

## 監査範囲

- 対象: 観点管理の一覧・編集画面
- 主利用環境: PC（1920×1080 / 1366×768、最小1024×768）
- ユーザー目標: 多数の観点を比較し、必要な観点を正確かつ効率的に編集・公開する
- 証跡: `01-current-1366x768.png`

## 現状評価

### 強み

- セット、一覧、編集の関係は同一画面内で把握できる。
- 検索、絞り込み、版タブ、公開操作の入口は見つけやすい。
- 既存WebSpec2Docの色、文字、ナビゲーション規則に沿っている。

### UX上の重大な問題

1. 編集領域が約350pxしかなく、長文の目的・推奨確認・根拠・条件式を扱う主作業に必要な幅がない。
2. 編集パネルが一覧幅を恒常的に奪い、一覧側でも観点名やカテゴリが省略される。比較と編集の両方が悪化している。
3. 1366×768では編集項目の大半が初期表示範囲外となり、保存前の全体確認が難しい。
4. 空の成功通知と「0件を選択」の一括操作バーが表示され、システム状態を誤って伝えている。
5. 編集開始、未保存、保存完了、競合、破棄の状態遷移が同じ狭い領域で競合し、操作の主従が弱い。

### アクセシビリティ上のリスク

- 狭い3カラムで読み順とフォーカス移動が長くなり、キーボード利用時に現在位置を見失いやすい。
- 省略表示が多く、テキストだけで内容を確認できない項目が生じる。
- 非表示予定の要素が表示されるため、支援技術へ不要な状態通知が行われる可能性がある。
- スクリーンショットだけでは読み上げ順序・名称計算・全キーボード経路は確定できないため、実装後のE2E確認が必要。

## 是正方針

- 一覧画面は検索・比較・一括操作へ集中させ、右側の常設編集パネルを廃止する。
- 行クリックまたは「観点を追加」で、PC向けの大きなモーダル編集画面を開く。
- 関連項目をグループ化し、重要項目と高度な条件を視覚的に分離する。
- 下部に破棄・保存を固定し、未保存・検証エラー・競合・保存完了を具体的な文言で表示する。
- Escape、フォーカストラップ、呼び出し元へのフォーカス復帰、Undoを必須とする。
- 保存後は一覧の同じ行・スクロール位置へ戻し、ユーザーの作業文脈を維持する。

## 受入基準

- 1920×1080と1366×768で主要編集項目と保存操作が同時に把握できる。
- 1024×768でもモーダル内だけが必要に応じてスクロールし、ページ全体に横スクロールが出ない。
- システム状態を空表示や色だけで示さない。
- エラーは原因と修正方法を対象項目の近くへ表示する。
- キーボードだけで開く、移動、保存、破棄、閉じる操作を完了できる。
