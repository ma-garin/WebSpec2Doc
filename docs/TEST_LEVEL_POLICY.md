# テストレベルの線引き（製品方針）

- 制定日: 2026-07-19
- ステータス: 確定（ロードマップ P2-4）
- 位置づけ: 本製品が「どこまでを担い、どこからを担わないか」の公式宣言

## 宣言

**WebSpec2Doc が担うのは、受け入れテスト〜システムテスト〜API 結合テストまでである。単体テストはスコープ外とする。**

| テストレベル | 本製品の扱い | 根拠 |
|---|---|---|
| 受け入れテスト | **担う** | 実画面の操作と観測で検証できる |
| システムテスト | **担う** | 同上。E2E の中核 |
| API 結合テスト | **担う** | クロール中に観測した呼び出しを起点に連結できる（P2-1） |
| 単体テスト | **担わない** | ソースリポジトリへの接続が前提となり、製品の性格が変わる |

## なぜ単体テストを担わないか

1. **必要な入力が違う**
   本製品の入力は「動いているWebサイトのURL」である。単体テストの生成にはソースコードそのものが要る。リポジトリ接続を前提にすると、対象は「Webサイト」ではなく「コードベース」になり、別製品になる。

2. **主張できる範囲が変わる**
   本製品は evidence-only 原則に立ち、「観測した事実」だけを主張する。単体テストは内部実装の意図を前提とするため、観測では代替できない。無理に踏み込むと、根拠のない主張を出す製品になる。

3. **先行製品も同じ線を引いている**
   mabl・testRigor など E2E 自動化の先行製品も E2E＋API までを守備範囲としており、単体テストへは踏み込んでいない。

## 利用者への説明文（そのまま使ってよい表現）

> WebSpec2Doc は、公開されている画面を実際に操作して観測し、受け入れ・システム・API 結合のテスト資産を作ります。
> ソースコードは読みません。したがって単体テストは作成しません。
> 単体テストは開発チームの実装作業として、別途ご用意ください。

## 将来の再評価条件

「リポジトリ接続オプション」として、以下がすべて満たされた場合に再評価する。

- 顧客側がソースリポジトリへの読み取り権限を提供できる契約形態が成立していること
- 観測ベースの主張境界と、コード解析ベースの主張境界を、成果物上で明確に区別できる設計が用意できること
- 単体テスト領域に既存ツール（各言語のテストフレームワーク付属機能）を上回る価値を出せる根拠があること

再評価するまでは、本方針を変更しない。
