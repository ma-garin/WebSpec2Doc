1. 現在地の整理と勝機

プロダクトの本質的価値

WebSpec2Doc は 「稼働中の Web システムの URL を渡すだけで、QA テスト設計に必要な全素材を自動生成する」唯一のツールである。

HTML/PDF/Excel/JSON

観点・ケース・条件
→
Playwright スクリプト
→
実行レポート

競合の穴 — 誰も解いていない問題

ツール得意なこと解決できないこと判定
Testim / Mablテスト実行・AI 修復仕様書・テスト観点の生成空白
TestRailテスト管理仕様抽出機能なし空白
Playwrightスクリプト実行基盤テスト設計支援なし空白
BrowserStackクロスブラウザ実行上流設計なし空白
WebSpec2Doc上流〜実行まで一気通貫—独自
最大の差別化軸: テスト設計の上流（テストベース生成）に特化した SaaS はグローバルでも存在しない。 テストエンジニアの工数の 50〜60% がテスト設計・仕様理解に費やされ、 アジャイル開発現場の 70%以上 で仕様書が存在しない or 更新されていない。

🔴 Phase 1: SaaS 基盤（Month 0–3・最優先）
2〜3 ヶ月
ローカル実行のままでは導入コストが高く失注する。「URL 入力 → 即使える」を実現することが全ての前提。
  • ユーザー認証（Auth0 / Firebase Auth・Google・GitHub OAuth）
  • 組織・プロジェクト管理（マルチテナント設計）
  • Stripe 課金統合（サブスクリプション + Usage-based）
  • クロールのバックグラウンドワーカー化（Celery + Redis）
  • クラウドデプロイ（AWS Fargate または Render.com）
  • 使用量メータリング（クロール数・ページ数・エクスポート数）
🔴 Phase 2: チーム機能（Month 2–4）
2 ヶ月
「個人ツール」から「チームツール」になることでチャーン激減・NRR 向上。
  • プロジェクト共有・権限管理（Owner / Editor / Viewer）
  • レポートへのコメント・メモ機能
  • Slack / Teams 通知（クロール完了・ドリフト検知時）
  • 顧客向けレポート URL 共有（認証付き外部共有リンク）
🟡 Phase 3: CI/CD インテグレーション（Month 4–6）
2 ヶ月
DevOps ワークフローに組み込まれると解約しにくくなる（スイッチングコスト向上）。
  • GitHub Actions 公式アクション（webspec2doc/action@v1）
  • GitLab CI テンプレート
  • Webhook API（外部トリガー対応）
  • REST API（プログラム的アクセス・OpenAPI 仕様書）
🟡 Phase 4: 既存ツール連携（Month 5–7）
2 ヶ月
連携先機能効果

7. 技術的競合優位（モート）

🗄️
モート 1: データネットワーク効果

各クロールの匿名化データを蓄積することで自動生成精度が向上。後発が追いつけないデータ資産になる。

  • 「EC サイトで頻出するテスト条件 TOP20」
  • 「認証フローの典型的な入力項目パターン」
🏭
モート 2: 業種別テンプレートライブラリ
業種テスト観点セット
EC サイトカート・決済・在庫
金融・銀行振込・セッション・暗号化
医療入力制限・アクセス制御
官公庁JIS X 8341 アクセシビリティ
📡
モート 3: AutoRun × 常時監視サービス

週 1 回の定期クロール → ドリフト検知 → Slack アラート → 自動テスト実行

Datadog がインフラ監視を独占したように、WebSpec2Doc が「Web 仕様の常時監視」というカテゴリを定義する。

9. リスクと対策

リスク影響度対策
Playwright 本家が Spec 生成機能を追加 高 業種テンプレート・LLM 統合で差別化を先行、データ資産で守る
大手 SaaS（Mabl 等）が日本語対応強化 中 検証会社との深い関係と日本語 UX・業種知識で守る
クロール対象サービスが拒否 低 User-Agent 管理、robots.txt 遵守、レート制限の徹底
セキュリティ懸念（認証情報管理） 高 SOC2 取得・暗号化徹底・ゼロ知識設計・auth.json の暗号化保存
開発リソース不足 高 Phase 1（課金基盤）のみに集中、残りは後回しにする
