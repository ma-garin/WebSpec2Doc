# WebSpec2Doc ユーザーガイド

URL を渡すだけで QA テストインプット文書を自動生成するツール。

---

## セットアップ（初回のみ）

```bash
# Python 3.12 が必要（3.13 は非対応）
python3.12 -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate

pip install -r requirements.txt
playwright install chromium
```

---

## GUI で使う（推奨）

```bash
python app.py
# → ブラウザが自動で http://127.0.0.1:8765 を開きます
```

### 4ステップの流れ

#### ステップ 1: URL 入力・解析

1. 「+ サイトを追加」をクリック
2. クロール対象の URL を入力（例: `https://example.com`）
3. **「画面分析」** ボタンを押す
4. 分析完了後、「N 件の画面を検出しました（経過時間）」と表示される
5. **「次へ →」** をクリック

#### ステップ 2: 条件設定

**画面の選択**
- 取得する画面にチェックを入れる（デフォルト：全選択）
- 不要な画面はチェックを外す

**ログインが必要なサイトの場合**
- 「要ログイン」バッジが付いた画面は画面リストの下部にグループ表示される
- 各「要ログイン」画面の直下に、**その画面専用の** ID/PASSWORD 入力フォームが自動表示される
- 検出されたログイン URL（例: `ログインURL: /login`）がフォーム上部に表示される
- **ID**（ユーザーID・メールアドレス）と **PASSWORD** を入力して **「ログイン」** を押す
- 自動ログイン後、認証後ページを含む再解析が実行される
- パスワードはセッション確立後に即破棄（保存されない）
- 複数の認証エリアがある場合は画面ごとに個別入力する

**その他のオプション**
- 「前回との差分を出力」をチェックすると仕様ドリフト検知が有効になる

**「解析開始」** をクリックしてステップ 3 へ進む。

#### ステップ 3: 実行

- クロールの進捗が 4 段階（受付 → ページ取得 → 解析 → 結果整理）でリアルタイム表示される
- ライブプレビューで取得中の画面を確認できる
- 完了すると **「レポートを見る →」** ボタンが表示される（実行時間付き）
- ボタンをクリックしてステップ 4 へ進む

#### ステップ 4: レポート

8 つのタブで成果物を確認できる。

| タブ | 内容 |
|------|------|
| **概要** | 画面インベントリ一覧（画面ID・タイトル・URL・フォーム数・入力項目数・遷移先） |
| **画面別仕様** | スクリーンショット（クリックで全画面表示）＋フォームカード＋テスト条件＋ページ要素（フォームなし画面） |
| **入力項目・テスト条件** | 全フィールドのテスト条件マトリクス（画面/型/必須フィルタ・検索・CSV 書き出し対応） |
| **設計** | テスト設計技法の推奨マトリクス（同値分割/境界値分析/決定表/状態遷移/分類木/PW法 等） |
| **技法詳細** | 各画面・各技法の機械導出サンプル値を確認できる詳細カード |
| **画面遷移図** | UML 遷移図を 3 種類のサブタブで表示：シーケンス図 / コミュニケーション図 / アクティビティ図 |
| **遷移表** | ISTQB 標準フォーマットの画面遷移表（イベント列にリンク先 URL パスを表示） |
| **履歴・差分** | クロール履歴と任意 2 時点の仕様ドリフト比較 |

**成果物のプレビュー** は各成果物カードの **「プレビュー」** ボタンで行う。アプリ内モーダルで表示されるため別ウィンドウは開かない。モーダルは「✕ 閉じる」ボタン・背景クリック・Escape キーで閉じる。別タブで開きたい場合は「↗ 新規タブ」をクリックする。

**エクスポート** はヘッダーの「↓ エクスポート」ボタンからいつでも行える。

| 形式 | 内容 |
|------|------|
| HTML レポート | 画面別仕様・テスト条件・スクリーンショット付き |
| PDF | HTML レポートの印刷・配布用 PDF 版 |
| JSON | 全データの構造化 JSON（自動化・CI 連携用） |
| Excel | spec.xlsx（テスト管理表への転記用） |
| Markdown（画面一覧） | screens.md |
| Markdown（フォーム） | forms.md |
| 遷移図（Mermaid） | transition.mmd |
| 差分レポート | diff_report.html |
| ZIP | 生成物一式をまとめてダウンロード |

## 再クロール（仕様ドリフト検知）

ダッシュボードの **「再クロール」** ボタンを押すと、前回の取得対象・設定を自動復元してステップ 2 から再開する。

- 設定を確認・変更してから「解析開始」
- 完了後のレポートの「履歴・差分」タブで前回との差分（追加/削除された画面・変更されたフォーム）を確認できる

## 品質観点を確認する

左サイドメニューの **「品質観点」** から、解析済みサイトの `report.json` を入力にして、CSV観点、アクセシビリティ、OWASP WSTG/ASVS、境界値、ペアワイズ、API契約、ビジュアル回帰候補を確認できる。

標準では外部 LLM API は呼び出さない。既存の `report.json` とQA観点CSVだけで生成する。QA観点には `data/qa_viewpoints_summary.csv` を使い、別CSVを使う場合は `QA_VIEWPOINTS_CSV` 環境変数で差し替えできる。

### 操作方法

1. 左サイドメニューの **「品質観点」** を開く
2. **対象サイト** を選択する
3. **「成果物を生成」** を押す
4. 左ペインの **成果物** から **「プレビュー」** ボタンで画面内確認、またはダウンロードを行う

### 生成される品質観点成果物

`output/{ドメイン名}/qa_process/` に保存される。

| ファイル | 内容 |
|---------|------|
| `quality_viewpoints.json` | 品質観点データ |
| `quality_viewpoints.html` | 品質観点HTMLレポート |

### トレーサビリティID

テスト設計とテストケースには、元画面・元フォーム・元入力項目へ戻れる Trace ID が付く。

| 例 | 意味 |
|----|------|
| `P001` | 画面ID |
| `P001-F01` | P001 の1つ目のフォーム |
| `P001-F01-I01` | P001 の1つ目のフォームの1つ目の入力項目 |
| `P001->P002` | P001 から P002 への画面遷移 |

## AutoRun（全自動テスト実行）

URL を入れるだけで、クロール → テスト設計 → Playwright スクリプト生成 → 実行 → 証跡までを通しで行う。
左メニューの **AutoRun** を開き、生成モードを選ぶ。

### 生成モード

| モード | 何を根拠にテストを作るか | 使いどころ |
|---|---|---|
| **URL駆動**（既定） | 実際に観測した画面 | 仕様書が無い・現物から起こしたい |
| **文書駆動** | 要件・仕様文書 × 観測した画面 | 仕様書があり、書かれた要件を漏れなく試したい |

### 文書駆動モードの手順

1. 生成モードで **「文書駆動」** を選ぶ
2. **対象URL** を入力する（先に入れないと文書をアップロードできない）
3. **要件・仕様文書** を追加する（複数可）
4. **パス選定基準** を選ぶ
   - 全画面を通る（頂点網羅）／全遷移を通る（エッジ網羅）／**全経路パターンを通る（プライムパス）**／指定画面まで到達
   - プライムパスは、ループを含む画面遷移を有限のテスト経路で漏れなく踏む基準です（エッジ網羅より強い）
5. 必要なら **「入力検証も観測する」** にチェック
   - フォームへ値を入れて検証メッセージだけを見る。**送信は一切しない**
6. 「開始」を押す

実行後、承認モーダルに「文書要件数 / 対応画面数 / 選定パス数 / カバー率」が表示される。
文書に書かれていても実際の画面で見つからなかった要件は、突合できなかったものとして残る
（勝手に埋めない）。

### 生成される主な成果物

| ファイル | 内容 |
|---|---|
| `document_mbt.json` | 要件と画面を結んだモデル・選定パス |
| `manual_procedures.md` / `.xlsx` | 手動テスト手順書（画面キャプチャ付き） |
| `test_data.json` / `.csv` | 実測の入力制約から導いたテストデータ |
| `pairwise_cases.json` | 選択肢の組合せを2因子網羅で圧縮したケース |
| `metamorphic_checks.json` | オラクル不在でも成り立つべき検証関係の候補 |
| `validation_observations.json` | 入力検証の観測結果（観測時のみ） |
| `evidence_pack.md` / `.html` | **検収提出用の実施証跡パック** |

未突合の要件（対応する実測画面が見つからなかったもの）には、テキスト類似度による
候補画面が根拠語つきで提示されます（自動リンクはせず、採用は人が判断します）。

### 要件を Gherkin（BDD）形式で書く

要件・仕様文書を **Gherkin 形式（`.feature`）** で用意すると、要件の抽出精度が上がります。
`docs/templates/requirements-template.feature` を雛形として使えます。日本語・英語の
両方に対応し、`@REQ-xxx` タグは要件IDとして保持され、「ならば」節は手順書の期待結果に
引き継がれます。

証跡パックには「いつ・どの環境で・何を実行し・結果がどうだったか」がまとまる。
材料が揃わなかった項目は「未取得」と明記される（空欄を埋めない）。

## CLI で使う

```bash
# 最小構成
python src/main.py --url https://example.com

# マルチビューポート（PC/タブレット/スマホの表示差分を文書化）
python src/main.py --url https://example.com --viewports desktop,tablet,mobile

# HTML レポート付き（推奨）
python src/main.py --url https://example.com --format md,html,excel,json

# 仕様ドリフト検知
python src/main.py --url https://example.com --compare

# 認証が必要なサイト（手動セッション保存）
python src/main.py --login https://example.com/login   # ブラウザでログイン後 Enter
python src/main.py --url https://example.com --auth auth.json
```

### オプション一覧

| オプション | デフォルト | 説明 |
|-----------|-----------|------|
| `--url` | 必須 | クロール対象 URL |
| `--depth` | `3` | リンクを辿る深さ |
| `--max-pages` | `50` | クロールする最大ページ数 |
| `--output` | `./output` | 出力先ディレクトリ |
| `--format` | `md` | 出力形式（`md,html,excel,pdf,json` をカンマ区切り） |
| `--compare` | off | 前回スナップショットとの差分を出力 |
| `--auth` | — | 保存済みセッション（auth.json）を指定 |
| `--viewports` | — | 複数画面幅で観測（`desktop,tablet,mobile`）。ビューポート間差分を文書化 |
| `--viewport-baseline` | `desktop` | ビューポート差分の比較基準 |

## 差分レポートのノイズを減らす

日付表示やアクセスカウンタのように毎回変わる箇所は、差分として毎回出てしまう。
`output/{ドメイン名}/diff_ignore.json` に無視ルールを置くと除外できる。

```json
{
  "rules": [
    {"kind": "regex",    "pattern": "^\\d{4}/\\d{2}/\\d{2}$", "note": "日付表示"},
    {"kind": "selector", "pattern": "#visitor-counter",       "note": "アクセスカウンタ"},
    {"kind": "field",    "pattern": "csrf_token",             "note": "トークン"},
    {"kind": "url",      "pattern": "/tmp/",                  "note": "一時ページ"}
  ]
}
```

除外した変更は**消えるのではなく別枠に退避**され、レポートに「除外N件（ルール別内訳）」として
必ず表示される。何を見なかったかが分かるようにするため。

差分には重要度（重大／警告／情報）と、そう判定した根拠が付く。
これはルールによる分類であり、変更が安全か危険かの判断ではない。

## 性能観測（Core Web Vitals）

クロール中に、各画面の表示性能を追加コストなしで観測します。レポートの「性能観測」節に
画面別の **LCP**（主要コンテンツ表示までの時間）・**CLS**（レイアウトのずれ）・**TTFB**
（サーバ初動）が出ます。

- これは**この環境での単一試行のラボ観測値**で、実利用者の体感や検索評価の値とは異なります。
  画面間の相対比較と、再クロール時の悪化検知に使ってください（合否は判定しません）。
- INP（操作への応答速度）は実ユーザーの入力が必要なため計測対象外です。

## レイアウトのはみ出し・重なりを見つける

`--viewports` でマルチビューポート観測を行うと、`viewport_report.html` の「レイアウト観測」節に、
その画面幅で**要素が画面外へはみ出した**箇所や**要素同士が重なった**箇所が出ます。
「スマホでボタンが見出しに隠れる」といった実害の発見に使えます（観測であり、バグ断定はしません）。

## 運用向け（社内サーバ展開時）

| 用途 | エンドポイント |
|---|---|
| APIリファレンス | `http://<host>:8765/api/v1/docs` |
| API仕様（機械可読） | `/api/v1/openapi.json` |
| 監視メトリクス（Prometheus形式） | `/metrics` |

`/metrics` ではクロールの成否・所要時間・スケジュール遅延・ジョブ滞留・通知成否を公開する。
「動いているか」ではなく「静かに壊れていないか」を見るための値。

## よくある質問

**Q: ログインが必要なページが取得できない**
→ ステップ 1 で「画面分析」を実行すると、認証が必要な画面は「要ログイン」バッジ付きで画面リスト下部にグループ表示されます。各画面の直下に表示される専用フォームに ID/PASSWORD を入力して「ログイン」を押してください。複数の認証エリアがある場合は画面ごとに個別入力します。

**Q: 画面遷移図がごちゃごちゃしている**
→ 共通ナビゲーション（ヘッダー/フッター）のリンクは自動的に除外されます。「画面遷移図」タブのサブタブ（シーケンス図 / コミュニケーション図 / アクティビティ図）を切り替えると、同じ遷移情報を別の視点で確認できます。

**Q: テスト設計技法の詳細なサンプル値を確認したい**
→ 「技法詳細」タブで、各画面・各技法の機械導出サンプル値（境界値・同値クラス・組合せ等）を確認できます。

**Q: 品質観点は追加料金がかかる？**
→ 標準のローカル生成ではかかりません。外部LLM/APIは呼び出しません。

**Q: 品質観点画面に対象サイトが出ない**
→ 先に「+ サイトを追加」からサイトを解析し、`report.json` を生成してください。旧データで `report.json` がない場合は再クロールが必要です。

**Q: 成果物を確認するたびに別ウィンドウが開く**
→ 各成果物カードの「プレビュー」ボタンを使うとアプリ内モーダルで表示されます（別ウィンドウは開きません）。モーダルは「✕ 閉じる」・背景クリック・Escape で閉じます。あえて別タブで開きたい場合は「↗ 新規タブ」をクリックしてください。

**Q: 品質観点で成果物が生成後に見つけにくい**
→ 品質観点画面は2ペインレイアウトで、左サイドバーに成果物リストが常時表示されます。生成後すぐ左サイドバーの成果物名をクリックするとプレビューできます。

**Q: 前回との差分を確認したい**
→ ダッシュボードから「再クロール」→ 「前回との差分を出力」にチェック → 「解析開始」。レポートの「履歴・差分」タブで確認できます。

**Q: 文書駆動モードで文書をアップロードできない**
→ 先に「対象URL」を入力してください。文書はサイト単位で保管するため、URL が未入力だとアップロード先が決まりません。

**Q: 文書駆動で「文書要件のうち一部が対応画面なし」と出る**
→ 文書に書かれていても、実際にクロールした画面の中に該当が見つからなかった要件です。未実装・別画面・クロール未到達のいずれかを人が判断してください。ツールは推測で埋めません。

**Q: 「入力検証も観測する」は本番サイトで安全か**
→ フォームに値を入れて検証メッセージを読むだけで、送信（submit / クリック / Enter）は行いません。値を入れる直前から通信を遮断するため、入力値が外部へ出ることもありません。

**Q: 検収用の報告書がほしい**
→ AutoRun 完了後に `qa_process/evidence_pack.html` が生成されます。実行日時・環境・ケース別結果・観点対応・画面キャプチャがまとまっています。ブラウザから印刷で PDF 化できます。

**Q: スマホで表示されない項目を洗い出したい**
→ `--viewports desktop,tablet,mobile` で実行すると `viewport_report.html` に「この画面幅では出なかった項目／この画面幅だけの遷移」が出ます。ただし片方に無いこと自体は不具合とは判定しません（意図的に隠している場合と区別できないため）。

**Q: `greenlet` のビルドが失敗する**
→ Python 3.13 は非対応です。`python3.12 -m venv venv` で Python 3.12 の仮想環境を作成してください。

**Q: P1 の画面分析がタイムアウトする**
→ 画面分析（discover）のタイムアウトは 180 秒です。ネットワークが遅い環境や大規模サイトでは超過する場合があります。対象 URL を絞るか、ステップ 2 の手動 URL 入力モードを使用してください。
