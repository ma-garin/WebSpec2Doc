## 2. $10M ARR の収益ロジック

### 2.1 なぜ $1M ARR 戦略の「10 倍」では到達できないか

07 文書の Starter 単独モデルでは算術的に到達不可能である。  
Starter（¥9,800/月 ≈ $65）× 12,800 社 × 12 ヶ月 = $10M だが、これは PLG だけで 12,000 社を獲得することを意味し、中小ツール SaaS では現実的でない。

**必要な戦略転換**: 「継続的に価値が出る機能」（常時監視・テスト資産の自動メンテナンス）で Team/Business の解約率を下げ、テスト設計エンジン＋外部連携で Enterprise 単価を引き上げる。

### 2.2 $10M ARR 達成モデル

```
プラン           | 契約数 | 月額（USD） | ARR 寄与
─────────────────────────────────────────────────────────
Starter          | 2,000社 |       $65 |  $1,560,000
Team             |   800社 |      $270 |  $2,592,000
Business         |   200社 |      $800 |  $1,920,000
Enterprise       |    40社 |    $5,000 |  $2,400,000
Usage-based 追加  |     —   |       —   |    $780,000
プロフェッショナル  |     —   |       —   |    $748,000
────────────────────────────────────────────────────
合計                                    $10,000,000 ARR
```

**到達前提**:  
- Starter → Team の転換率 8%（業界平均 5〜10%）
- Team → Business の転換率 15%（差別化機能実装後）
- Business → Enterprise のアップセル率 10%（専任 CSM + PoC 支援）
- チャーン月次 < 1.5%（NRR 115%+ = 拡張収益がチャーンを上回る）

### 2.3 チャーンを下げる「ロックイン」設計の原則

| ロックイン手法 | 実現する機能 | 解約困難になる理由 |
|---|---|---|
| ワークフロー組み込み | CI/CD 連携・Webhook | リリースパイプラインに WebSpec2Doc が入ると抜けられない |
| データ蓄積 | スナップショット履歴・テスト資産 | 過去データが企業の QA 資産になる |
| チーム定着 | レビュー＆確定ワークフロー | 複数人の日常業務フローになる |
| 外部連携 | TestRail/Jira 同期 | 他ツールとの連携が切れると業務が止まる |

### 3.1 現在の「価値提供のエッジ」と「穴」

```
現在の価値提供:
  URL → [クロール] → [仕様抽出] → [テスト条件生成(ルールベース)] → レポート出力
  
$10M に必要な価値提供:

                                                               ↓
                              TestRail/Jira へ同期 ← [レビュー確定ワークフロー]
```

### 3.2 機能ギャップと収益への紐付け

| 機能ギャップ | 欠如した場合の収益影響 | 対応ピラー |
|---|---|---|
| SPA・状態クロール非対応 | 実案件で「画面が拾えない」と即解約（Churn 直結） | P0-A |

### P0-A クロールカバレッジ深化

**位置付け**: 商品力の土台。クロール精度が低いと全ての価値が崩壊する。  
**対象プラン**: 全プラン（基本機能として必須）  
**収益寄与**: Starter→Team 転換率向上、Enterprise PoC 通過率向上

#### P0-A-1: SPA クライアントルート対応

**対象モジュール**: `src/crawler/page_crawler.py`

現状の `page_crawler.py` は `crawl_site()` 内で URL ベースで既訪問判定しているため、`pushState` / `hashchange` によるルート変化を新規ページとして認識できない。

**実装すべきアルゴリズム**:

```python
# page_crawler.py の crawl_site() 拡張イメージ
async def _handle_route_change(page, prev_url, new_url):
    """pushState / hashchange 後の URL 変化を新規ページとして登録"""
    if _normalize(new_url) != _normalize(prev_url):
        await page.wait_for_load_state("networkidle")
        return PageData(url=new_url, ...)

# page.on("framenavigated") ハンドラで検知
page.on("framenavigated", lambda frame: _handle_route_change(...))
```

**新規追加画面**:
- ウィザード設定「SPA モード ON/OFF（デフォルト OFF・誤爆防止）」
- ページ一覧タブに「SPA ルート」バッジ付き行を追加

#### P0-A-2: 状態認識クロール（同一 URL 別状態を別画面として記録）

**対象モジュール**: `src/crawler/page_crawler.py`, `src/crawler/link_extractor.py`

モーダル・タブ・アコーディオン・条件表示は同一 URL だが、テスト対象としては別画面である。  
DOM シグネチャ（表示中の主要要素のハッシュ）で「状態」を識別し、`PageData` に `state_id` を付与する。

**実装すべきアルゴリズム**:

```python
# link_extractor.py に追加
def compute_dom_signature(page_html: str) -> str:
    """表示中のインタラクティブ要素の構造ハッシュ。
    モーダル開閉・タブ切替で変化する最小セットを対象とする。"""
    soup = BeautifulSoup(page_html, "html.parser")
    visible_ids = [el.get("id", "") for el in soup.find_all(
        lambda t: t.get("aria-hidden") != "true" and t.name in
        {"dialog", "form", "section", "[role='tabpanel']"}
    )]
    return hashlib.sha1("|".join(visible_ids).encode()).hexdigest()[:8]
```

クリック可能要素（ボタン・タブ・アコーディオン）を探索して状態を列挙し、各状態でスクリーンショット＋フォーム抽出を行う。

**必要な画面変更**:
- レポートの「画面一覧」に「状態 (State)」列を追加
- 遷移図（`src/generator/mermaid_generator.py`）でサブステートを `state` ノードとして表現

#### P0-A-3: マルチステップフロー実行

**対象モジュール**: `src/analyzer/test_conditions.py`, `src/crawler/page_crawler.py`

`test_conditions.py` が導出する境界値・同値分割のダミー値を使い、フォームを実際に送信して確認画面・エラー画面を取得する。

async def crawl_form_flow(page, form_data: FormData, dry_run: bool = True) -> list[PageData]:
    """フォームにダミー値を入力し送信後の画面を取得。
    dry_run=True: submit をインターセプトしてキャンセル（確認画面は取得、実際の送信なし）。"""
    ...
```

**新規 GUI 設定**:
- ウィザード ステップ 2「フォーム送信クロール（beta）」チェックボックス
- dry-run モードのデフォルト ON 表示

@dataclass(frozen=True)
class FieldDiff:

    after: str | None
    severity: Literal["breaking", "warning", "info"]
```

#### P1-A-2: デシジョンテーブル自動導出

**対象モジュール**: `src/analyzer/test_conditions.py`（拡張）

複数フィールドの入力値の組み合わせが出力・動作を変える場合（条件分岐）を、遷移グラフ（`src/graph/transition_graph.py`）から推論しデシジョンテーブルとして出力する。

**導出ロジック**:
- 遷移グラフのエッジに `condition` 属性（入力値・バリデーション条件）が付いている経路を抽出
- 同一フォームから分岐する複数の遷移先を「条件」として整理
- デシジョンテーブル（条件 × アクション）形式に変換

#### P1-A-3: 状態遷移テスト（N スイッチカバレッジ）

**対象モジュール**: `src/graph/transition_graph.py`（拡張）

遷移グラフから W メソッド / DS メソッドに基づいた状態遷移テストケースを自動導出する。

| カバレッジ基準 | 説明 | 生成ケース数の目安 |
|---|---|---|
| 0 スイッチ（全状態） | 全ページを 1 度は訪問 | ページ数と同程度 |
| 1 スイッチ（全遷移） | 全リンク/フォーム送信を 1 度は通過 | 遷移エッジ数 |
| 2 スイッチ（全遷移対） | 連続する 2 遷移の組み合わせ | エッジ数の 2 乗程度（上限付き） |

### P1-B 生成テスト実行の実用化

**位置付け**: 「生成した spec.ts が CI で壊れる」問題を解消し、Enterprise の CI/CD 組み込みを実現。  
**対象プラン**: Team（基本実行）・Business（self-healing + 失敗分類）・Enterprise（SLA 付き実行）  
**収益寄与**: スイッチングコスト向上・Enterprise の CI 組み込みで解約不可状態

#### P1-B-1: アサーション強化

**対象モジュール**: `web/services/spec_ts_generator.py`

現状の生成 spec.ts はページ遷移後の `body visible` のみをアサートする。  
以下のアサーションを追加生成する。

| アサーション種別 | 生成ロジック |
|---|---|
| URL 遷移検証 | `expect(page).toHaveURL(...)` — 遷移グラフのエッジから生成 |
| バリデーションメッセージ検証 | `expect(page.locator('[role=alert]')).toContainText(...)` — test_conditions の期待エラー文から生成 |
| フォーム送信後の状態検証 | 確認画面・完了画面 URL への遷移をアサート |
| エラー状態の画面検証 | バリデーションエラー時の `aria-invalid` 属性変化をアサート |

#### P1-B-2: self-healing ロケータ

**対象モジュール**: `web/services/spec_ts_generator.py`, `src/generator/json_reporter.py`

`report.json` には各要素の複数ロケータ候補（`id` / `aria-label` / `data-testid` / CSS セレクタ / XPath）がすでに格納されている。  
生成する spec.ts で Playwright の locator チェーンをフォールバック順に組む。

```typescript
// 生成コードのイメージ
const emailField = page.locator([
  '[data-testid="email"]',
  '[aria-label="メールアドレス"]',
  'input[type="email"]',
  '#email',
].join(', ')).first();
```

CI で DOM が変化した場合、最初のロケータが失敗しても次の候補にフォールバックし、テスト自体は通過する。  
失敗したロケータはログに記録し、次回 spec.ts 再生成時に優先度を下げる。

#### P1-B-3: 失敗原因分類

**新規モジュール**: `web/services/failure_classifier.py`

Playwright 実行結果（`playwright_executor.py` から取得）を解析し、失敗を分類する。

```python
FailureType = Literal[
    "app_change",    # アプリの仕様変更（差分検知と照合）
    "test_rot",      # ロケータ変化・タイミング問題（アプリは正常）
    "env_issue",     # ネットワーク・認証切れ・サーバーエラー
]
```

分類結果を `diff_report.html` の「実行結果」タブに表示し、`app_change` は自動で差分レポートと紐付ける。
