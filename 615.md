## 2. CX（継続利用・顧客体験）

**C1 — オンボーディング（P1/低）**
論拠: 削除済タスク `codex-phase2-c-onboarding.md` の痕跡があり、初回体験が未確立。「URLを渡すだけ」価値が初回で伝わらないと離脱する。
提案: 初回起動時に内蔵サンプル（または `example.com`）でワンクリックのデモ実行を提供し、成果物8タブを体験させる。

**C2 — 定期再クロール自動化（P1/中）★継続利用の核**
論拠: `plan.txt` は「一度使ったら定期的に使い続ける」を楔と定義。`web/services/scheduler.py`・`web/routes/schedule.py` は存在するが、UIからの定期実行〜結果蓄積の体験が弱い。
提案: サイトごとに cron 的スケジュール（毎日/毎週）を設定 → バックグラウンド再クロール → 差分を履歴に自動蓄積。**ジョブ永続化（O2）とセット**で初めて成立する点に注意。

**C3 — ドリフト通知（P1/中）★ROIの可視化**
論拠: `plan.txt` の楔説明に「差分レポート＋Slack/Teams通知」、`web/services/notifier.py` 実装あり。`improve-p3-drift-notification.md` は画面内サマリー差分まで。
提案: 再クロール完了時に Webhook で「新規画面N / フォーム変更M / 推奨技法変化K」を外部通知。これが「変更検知→テスト範囲特定」のROIを顧客に伝える最短経路。

**C4 — 横断ダッシュボード（P2/中）**
論拠: `src/registry/site_registry.py` で複数サイトを管理できるが、`view-dashboard.html` は単一サイト中心。
提案: 登録サイト一覧に「最終クロール日時 / 直近の差分件数 / 要再テストフラグ」を並べ、運用者が朝に一覧で異常を発見できる体験にする。

**C5 — 失敗リカバリ導線（P2/中）**
論拠: `feature_contracts.yml` に `session_expired` / `partial_checkpoint` / `cancel` の failure_mode が定義済だが、UI側の「再開ボタン」導線が弱い。
提案: セッション期限切れ検出時に再ログインモーダルへ直結、部分クロール失敗時はチェックポイントからの再開を提示。`functional-integrity` の recovery 要件を体験として満たす。

## 3. アルゴリズム

| # | 提案 | 優先 | 難易 |
|---|------|------|------|
| A1 | 画面の正規化/同一性判定の精度向上（差分誤検知の低減） | P1 | 高 |
| A2 | 共通ナビ判定をDOM構造ベースに強化 | P2 | 中 |
| A3 | ドリフト差分のリスクスコアリングと再テスト範囲提案 | P2 | 高 |
| A4 | テスト技法推奨の組合せ爆発検出（CT/pairwise） | P2 | 中 |
| A5 | 大規模サイト向けクロール予算制御と優先度付け | P3 | 高 |

**A1 — 正規化/同一性判定（P1/高）★差分品質の根幹**
論拠: `src/analyzer/canonicalizer.py` 存在。`feature_contracts.yml` に `session_expired_false_diff`（誤差分）という failure_mode が明記＝既知リスク。URLの動的ID/クエリ揺れで同一画面が別画面と判定されると差分が信頼されなくなる。
提案: URLパラメータ正規化＋ページ指紋（DOM構造ハッシュ/主要フォーム）で同一性を判定し、再クロール差分のノイズを削減。**C2/C3の通知価値はここの精度に依存**。

**A2 — 共通ナビ判定の強化（P2/中）**
論拠: `CONTEXT.md`「全ページの50%以上から出るリンクは共通ナビと判定して非表示」。単純な出現率閾値はサイト規模で誤判定しやすい。
提案: `<header>/<nav>/<footer>` 等の構造的位置と出現率を併用したスコアで判定。閾値ハードコードを設定可能にする（coding-style: ハードコード禁止）。

**A3 — 差分のリスクスコア化（P2/高）**
論拠: `src/diff/impact_analyzer.py` 存在。現状の差分は「追加/削除/変更」の列挙が中心。
提案: 変更を「破壊的（フォーム必須項目変更・削除）/中（要素追加）/軽微（テキスト）」に分類しスコア化、影響画面から **優先再テスト対象**を提案。Spec Supplier として AIエージェントに渡す価値が上がる。

**A4 — 技法推奨の組合せ検出（P2/中）**
論拠: `src/analyzer/technique_recommender.py`(335行)。`improve-p3-technique-recommender.md` で action URL ブースト着手中だが、組合せ技法（CT/pairwise）の発火条件が弱い。
提案: 同一フォーム内の独立した選択肢（select/checkbox/radio）数から組合せ爆発を検出し、件数とともに pairwise を推奨。`ct/comb` キーの推奨根拠を可視化。

**A5 — クロール予算制御（P3/高）**
論拠: `src/crawler/parallel_crawler.py` / `page_crawler.py`(785行)。大規模サイトで深さ・件数が膨らむと時間/誤検知が増える。
提案: 幅優先＋ページ重要度（フォーム有無・被リンク数）で優先キューを構成し、上限到達時は「重要画面を優先的に残す」賢い打ち切りにする。

## 5. その他（アーキテクチャ / 品質 / セキュリティ）

| # | 提案 | 優先 | 難易 |

| O1 | 永続化を SQLite 化（snapshot/registry/job） | P2 | 高 |
| O2 | ジョブキューの堅牢化（再起動耐性・重複実行防止） | P2 | 高 |
| O3 | 大ファイルの分割リファクタ（800行規約遵守） | P3 | 中 |
| O4 | データ外部送信ゼロ保証の明文化＋自動テスト | P2 | 中 |
| O5 | 観測性向上（構造化ログ・失敗分類の集約） | P3 | 中 |

**O1 — SQLite化（P2/高）**
論拠: 永続化はファイルベース（`output/{domain}/...` JSON、`data/` にCSV、DBファイルなし）。継続監視（C2/C4）で履歴が増えると一覧/集計/差分照会がファイル走査になり破綻する。
提案: snapshot/site_registry/job を SQLite に集約（オンプレ前提なので外部DB不要）。差分・横断ダッシュボードのクエリ基盤になる。

**O2 — ジョブキュー堅牢化（P2/高）**
論拠: `web/services/job_queue.py` は `_JOBS: dict` のインメモリ＋スレッド管理。プロセス再起動でジョブ消失、定期実行（C2）には不向き。
提案: ジョブ状態を永続化（O1基盤）し、同一サイトの重複クロールをロックで防止。スケジュール実行の信頼性を担保。

**O3 — 大ファイル分割（P3/中）**
論拠: coding-style 規約「800行上限」。`src/crawler/page_crawler.py`(785) はほぼ上限、`static/js/autorun.js`(768)・`web/routes/auto_run.py`(670) も肥大。
提案: 責務単位で分割（例: page_crawler→収集/抽出/スクショ）。U3のAutoRun改修と同時に `autorun.js` を分割。

**O4 — 外部送信ゼロ保証（P2/中）**
論拠: `plan.txt` 収益核は「クライアント情報を外部送信できない第三者検証会社向けオンプレ」。`src/crawler/url_safety.py`・`web/validation.py`・`web/security.py`・削除済 `allow-local` タスク群あり。保証が暗黙的。
提案: 「APIキー未設定時は一切の外部通信をしない」ことを自動テストで担保し、SSRF/ローカルアクセス制御を継続強化。販売時の信頼根拠を技術で裏付ける。

**O5 — 観測性（P3/中）**
論拠: `web/services/failure_classifier.py` で失敗分類の素地あり。運用（定期実行）では失敗の傾向把握が重要。
提案: 構造化ログ＋失敗分類をダッシュボードに集約し、「どのサイトでどの失敗が多いか」を可視化。運用品質と継続改善のループを作る。

## 6. 視座を広げた追加施策（システム完成度の引き上げ）

前半の25件は「機能・画面・アルゴリズム」の改善だった。ここでは一段上の視座＝**プロダクトとして売れる/運用に乗る/品質が劣化しない**ための横断施策を、5テーマ15件追加する。これらは個別機能ではなく「製品の土台」を底上げする。

### 6.1 プロダクト品質の自己実証（独自性・最重要）

| # | 提案 | 優先 | 難易 |
|---|------|------|------|
| Q1 | スペック品質フィードバックループ | P2 | 高 |
| Q2 | ジェネレータのゴールデン回帰テスト | P1 | 中 |
| Q3 | functional-integrity ゲートのCI強制 | P1 | 低 |

**Q1 — スペック品質フィードバックループ（P2/高）★この製品にしかできない検証**
論拠: AutoRun は spec.ts を生成して**実際にテスト実行**する（`web/services/playwright_executor.py`、状態 `running_tests`）。この実行結果が捨てられている。
提案: 生成テストの pass/fail/flaky を画面・技法単位で蓄積し、「生成スペックが実環境で通る率」を品質指標化。ロケータ生成（F1）や技法推奨（A4）に還元する。**「自分が出した仕様の正しさを自分で測れる」**のは競合（Screaming Frog/codegen）に無い決定的強み。

**Q2 — ジェネレータのゴールデン回帰（P1/中）**
論拠: 2026-06-14 に「技法タブがサマリーモードで消える」生成品質の退行が発生（`improve-p1-e2e-smoke.md`）。E2Eは入口を守るが、**生成物の中身の劣化**は検知できていない。
提案: 固定サンプルサイト集に対する生成物（report.json/技法/遷移表）のスナップショットを基準として保存し、差分が出たらレビュー必須にする。生成品質の「静かな劣化」を機械的に止める。

**Q3 — 品質ゲートのCI強制（P1/低）**
論拠: `CLAUDE.md`/`.claude/rules/functional-integrity.md` が `quality_harness.py`・`make test`・`make verify-ui` を完了条件にしているが、`.githooks` 任せで pre-merge 強制が弱い。
提案: GitHub Actions でこの3つを必須チェック化し、未通過はマージ不可に。ルールを「人の運用」から「仕組み」へ移す。

### 6.2 データガバナンス／コンプライアンス（オンプレ収益の前提）

| # | 提案 | 優先 | 難易 |
|---|------|------|------|
| D1 | 出力データのライフサイクル管理（保持期間/削除） | P2 | 中 |
| D2 | 入力値・PII のマスキングと操作監査ログ | P2 | 中 |
| D3 | 「外部送信ゼロ」のネットワーク遮断実証 | P2 | 中 |

**D1 — データライフサイクル（P2/中）**
論拠: `output/{domain}/` に顧客サイトのスクショ・仕様が永続。`plan.txt` の顧客は第三者検証会社＝クライアント情報の取扱いが契約事項。現状、保持期間や削除UIがない。
提案: サイト単位の保持期間設定・期限切れ自動削除・「このサイトのデータを完全削除」UI（CRUD-Dは確認付き）。データを溜めっぱなしにしない仕組みを標準装備。

**D2 — マスキングと監査（P2/中）**
論拠: 自動ログインでID/PASSWORDを扱う（`src/crawler/auto_login.py`）。パスワードは破棄されるが、スクショ・ログに入力値や個人情報が写り込む懸念。
提案: スクショ/ログの入力値マスキング設定と、「誰がいつどのサイトをクロール/削除したか」の操作監査ログ。Enterprise調達のセキュリティ要件に直結。

**D3 — 外部送信ゼロ実証（P2/中）**
論拠: `plan.txt` 収益核＝「外部送信できない現場向けオンプレ」。`url_safety.py`/`web/security.py` はあるが、保証が暗黙的。前半 O4 を制度化する。
提案: ネットワーク遮断環境での全機能動作を自動テスト化し、「APIキー無し時は一切外部通信しない」をリリースごとに証跡化。販売資料に添付できる信頼根拠にする。

### 6.3 配布・運用・環境堅牢性（オンプレ導入の容易さ）

## 7. テスト設計の自動化と論理化（テスターの設計業務を肩代わりする）★戦略の本丸

**ご質問への結論: 現状の615案には「設計業務を奪う」施策は不足している。**
前半の F1（spec.ts）・A4（組合せ"検出"）・Q1（品質ループ）は断片で、全体としては今も「人間の設計者に**材料を渡す**（Spec Supplier）」発想に留まる。テスター/テストエンジニアの**設計そのもの**を置き換えるには、各技法を「推奨」から「**具体値つき完全テストケースの生成**」へ引き上げ、生成物を**論理的に保証**（カバレッジ・トレース・根拠）する必要がある。これを独立した本丸トラックとして提案する。

### 自律度ラダー（製品が登るべき段）

```
L0 材料供給        画面仕様・推奨技法を渡す（≒現状）
L1 ケース雛形       技法ごとの空欄ケースを並べる（一部実装済）
L2 完全ケース生成   入力値・期待結果・前提まで埋まった実行可能ケース ← 奪い始める
L3 論理保証        カバレッジ% / トレース / 最小化 / 根拠つき   ← 設計者をレビュー役に
L4 自己実証        生成→実行→pass率で設計品質を自己測定（Q1）   ← 設計者不要に近づく
```

「奪う」の正体は L2→L3 への到達。L3 まで行くとテスターの役割は**設計者から「機械が設計した一式の承認者」**へシフトする。以下 7.1（技法を生成エンジン化）と 7.2（論理保証基盤）で構成する。

### 7.1 技法エンジン群（「推奨」→「生成」へ）

| # | 提案 | 優先 | 難易 |
|---|------|------|------|
| T0 | フォーム制約の本格抽出（境界値生成の前提） | P1 | 中 |
| T1 | 同値分割＋境界値の自動導出（具体値生成） | P1 | 中 |
| T2 | デシジョンテーブル生成（条件→ルール→ケース） | P2 | 高 |
| T3 | 状態遷移テスト（0/1-switch・不正遷移） | P1 | 高 |
| T4 | ペアワイズ/組合せの実生成（IPOG等） | P2 | 中 |
| T5 | 分類木法（CTM）/原因結果グラフ | P3 | 高 |

**T0 — 制約抽出の本格化（P1/中）★全ての論理設計の前提**
論拠: `src/analyzer/form_analyzer.py` は現状 `field_type`/`required` しか抽出していない（確認済）。`min`/`max`/`maxlength`/`minlength`/`pattern`/`step`/`type=email|number|date` 等を取っていない＝**境界値もクラスも論理的に導けない**。
提案: HTML5制約属性・`aria-*`・周辺ラベル/プレースホルダから入力ドメイン（型・範囲・書式・必須）を構造化抽出。これが T1〜T5 全ての入力になる。

**T1 — 同値分割＋境界値の具体値生成（P1/中）**
論拠: `technique_recommender.py` は技法を推奨するが、`ep/bva` の**具体値**を機械生成していない。
提案: 制約（T0）から有効/無効同値クラスを列挙し、各境界に対し `min-1 / min / min+1 / max-1 / max / max+1`、長さ境界、書式合致/非合致を**実値**で生成。例: `年齢[18-120]` →無効`17`,有効`18`,`19`…有効`119`,`120`,無効`121`。これだけでテスターの単純設計の大半を代替する。

**T2 — デシジョンテーブル生成（P2/高）**
論拠: 複数フィールドの相互依存・バリデーション分岐（例: クーポン有×在庫無→不可）は人手設計の中核。`test_conditions.py` に `product`（直積）の素地はあるが圧縮・ルール化はない。
提案: 条件（必須/相互依存/分岐）を抽出→条件の組合せ表を構築→don't-care で圧縮→各ルールを1ケースへ写像。表とケースの双方向リンクで根拠を明示。

**T3 — 状態遷移テスト（P1/高）★既存資産が活きる**
論拠: `src/graph/transition_graph.py` で遷移グラフを既に構築済（`CONTEXT.md` の遷移図/遷移表）。これは**状態遷移テストの形式モデルそのもの**なのに、テストパス生成に使われていない。
提案: グラフから Chow の **0-switch（全遷移1回）/1-switch（連続2遷移）= N-switch** カバレッジを満たすテストパス（遷移木）を自動生成。さらに定義されていない遷移＝**不正遷移（sneak path）**を列挙して異常系ケース化。遷移表のNoとテストパスを対応付ける。

**T4 — ペアワイズ/組合せの実生成（P2/中）**
論拠: A4は組合せ爆発の「検出」止まり。`test_conditions.py` に `_MAX_PAIRWISE_FIELDS` 等の素地はあるが、被覆保証されたケース集合の生成アルゴリズムがない。
提案: 因子・水準から **IPOG/IPOG-F 等で pairwise（必要なら n-wise）** のケース集合を生成し、「全ペア網羅・◯ケースに削減」を明示。制約（禁則組合せ）にも対応。

**T5 — 分類木法（CTM）/原因結果グラフ（P3/高）**
論拠: 大きな入力空間の構造化は熟練テスターの暗黙知。標準技法として形式化できる。
提案: 入力空間を分類（観点）×クラスの木に整理し、最小組合せ網羅でケース導出。原因結果グラフ（論理関係→決定表）も中長期で。可視化して設計レビューに供する。

### 7.2 設計を論理的に保証する基盤（L2→L3 への橋）

| # | 提案 | 優先 | 難易 |
|---|------|------|------|
| L1 | カバレッジモデルと到達率の測定・表示 | P1 | 中 |
| L2 | 双方向トレーサビリティの一気通貫化 | P1 | 中 |
| L3 | 期待結果（オラクル）の半自動導出＋未確定の明示 | P2 | 高 |
| L4 | ネガティブ/異常系の体系的導出 | P2 | 中 |
| L5 | テストケース最小化（集合被覆）と冗長検出 | P2 | 中 |
| L6 | リスクベース優先度付け（影響×重要度×複雑度） | P2 | 中 |
| L7 | 設計判断の根拠提示（説明可能なテスト設計） | P1 | 中 |

**L1 — カバレッジモデルと測定（P1/中）★「論理的」の中核**
論拠: 現状「推奨技法バッジ」はあるが、**網羅できているかの定量指標がない**。論理的設計とは「何を網羅し何を網羅していないか」が言える状態。
提案: 技法ごとに網羅基準を定義（EP=全クラス、BVA=全境界、DT=全ルール、ST=全0/1-switch、CT=全ペア）し、生成ケースに対する**カバレッジ%と未達リスト**を算出・表示。「この画面はBVA 100% / DT 4/5ルール（残: 在庫切れ×クーポン）」のように出す。

**L2 — 双方向トレーサビリティ（P1/中）★既存資産を拡張**
論拠: `web/services/traceability.py` に `RequirementLink`（coverage: covered/partial/uncovered, coverage_rate）の土台が既にある。これを設計連鎖に広げる。
提案: **画面要素/要件 → テスト条件 → 技法 → テストケース → テストデータ → 自動テスト(spec.ts)** を一意IDで双方向リンク。「この境界値ケースはどの制約由来で、どのspec.tsに対応するか」を辿れる＝監査可能な設計。

**L3 — 期待結果（オラクル）の半自動導出（P2/高）★最難関＝差別化点**
論拠: テスター代替の最大の壁は「期待結果が分からない」オラクル問題。だが**ルール由来の期待は導出可能**。
提案: バリデーション制約から「無効入力→該当エラー期待」、リダイレクト/HTTP/遷移グラフから「成功遷移先」、必須漏れ→「送信不可」を機械導出。**導出不能な箇所は『期待結果: 要人手判断』と明示**（誠実なオラクル管理）。これにより人手は「全設計」でなく「不確定な期待値の確定」だけに集中できる。

**L4 — ネガティブ/異常系の体系生成（P2/中）**
論拠: 異常系は網羅漏れが起きやすく、テスターの腕が出る領域。制約があれば機械化できる。
提案: 各制約の違反パターン（型違反・範囲外・書式不正・必須漏れ・長さ超過・インジェクション文字）、認証/認可の不正系（未ログイン到達・他者リソース）、状態の不正遷移（T3）を技法から網羅生成。

**L5 — テストケース最小化（P2/中）**
論拠: 生成ケースは放置すると爆発し、保守不能になる（テスターが嫌うのはこれ）。
提案: 同一カバレッジ（L1）を満たす**最小ケース集合を集合被覆（グリーディ）で算出**し、冗長ケースを検出・統合。「品質を落とさず◯件→◯件に削減」を提示。

**L6 — リスクベース優先度付け（P2/中）★既存資産が活きる**
論拠: `src/diff/impact_analyzer.py`（変更影響）＋ action URL の業務重要度（A3/`improve-p3-technique-recommender`）が既にある。
提案: 各ケースに `発生影響(impact) × 業務重要度 × 入力複雑度` のリスクスコアを付与し、実行順・スモーク選別（AutoRunの「実行方針」）に直結。論理的な「どこから/どこまでテストするか」を根拠つきで示す。

**L7 — 説明可能なテスト設計（P1/中）★レビュー役へのシフトを成立させる**
論拠: 機械生成を人が承認するには「**なぜこのケースが要るのか**」が不可欠。根拠が無ければ承認も削除もできず、結局テスターが作り直す。
提案: 各ケースに導出根拠を付与（例: 「BVA: 年齢上限120の上側境界」「DT: ルール3（会員×在庫有）」「ST: 1-switch ログイン→検索→購入」）。これにより成果物が**レビュー可能＝設計者を承認者に変える**。L1〜L6の全出力に共通フィールドとして持たせる。

> **このトラックの肝**: T0→T1→T3→L1→L2→L7 を最小一周させると、「制約から具体値ケースを生成し、カバレッジと根拠つきで、遷移モデルまで含めて出す」という**設計者の中核業務の代替**が動き出す。Q1（自己実証）と繋げば、出した設計の正しさを実行で測る L4 自律度に届く。
