# WebSpec2Doc 実機デモ台本

製品機能の実演手順。同梱デモサイト **DemoMart** を対象に、
外部ネットワーク・OpenAI API キーなしで完走する。

## 事前準備（発表前に済ませておく）

```bash
make demo
```

- デモサイト（DemoMart・現行）: http://127.0.0.1:8766/
- デモサイト（DemoMart・新側 site_v2）: http://127.0.0.1:8767/（移行検証デモ用に併走起動）
- WebSpec2Doc 本体: http://127.0.0.1:8765/
- ネットワーク遮断環境でも動作（顧客データを外部送信しないことの証明）
- 失敗保険: `docs/demo/sample_output/report.html` を別タブで開いておく。
  ライブ生成が失敗しても、この生成済みレポートで説明を続行する。

## 10分版（質疑込み・詳細デモ）

5分版に加えて、以下を挟む:

1. **認証ウォール検出**（1:15 付近）: ログイン画面が「要ログイン」として検出される様子。「認証の必要な画面も、ログインすれば認証後ページまで解析します」
2. **画面状態の検出**（2:00 付近）: マイページのモーダル・タブ・アコーディオンが「別の画面状態」として記録されること。「同じ URL でも状態が違えば別画面として扱う=SPA・動的サイトに対応」
3. **バリデーション実測**（2:20 付近）: 必須フォームを空送信したときのブラウザ実測メッセージが「期待結果（実測）」に転記される様子。「テスト条件の期待結果を、人手ではなく実測で埋めます」
   - 注: メッセージ言語はブラウザのロケールに従う（日本語 OS では日本語で表示）
4. **SPA 遷移捕捉**（3:00 付近）: キャンペーンページの pushState/hashchange が遷移グラフに反映される様子

## 移行検証デモ（現新比較・SPEC-3-5）

Sprint B/C で増えた「現新比較モード」（OS移行・リプレイス検証支援）を単独で見せる短尺デモ。
`make demo` で 8766（現行）・8767（新 site_v2）が併走起動している前提。

| 時間目安 | 操作 | 話すこと（要点） |
|---|---|---|
| 0:00 | ターミナルで比較実行を叩く（or 事前生成した `compare_*/comparison.html` を開く）: `python src/main.py --compare-old-urls http://127.0.0.1:8766/ --compare-new-urls http://127.0.0.1:8767/` | 「移行前後の 2 バージョンを実測クロールし、画面を自動で対応付けます」 |
| 0:45 | comparison.html の「網羅性サマリ」タイル | 「まず**どこまで比較できたか**を明示します。対応付け n 組・現行のみ m・新のみ k・検査できなかったリンク j 件。**断定せず未確認は未確認と言う**のが売りです」 |
| 1:30 | 4 分類（表示崩れ/文字化け・意味消失/理解不可能/操作不可）セクション | 「移行で壊れがちな不具合をカテゴリ別に自動分類。分類できない差分は『未分類（要確認）』として握りつぶしません」 |
| 2:15 | 「対応画面が見つからなかった画面」セクション | 「新規追加・削除画面も明示するので、移行漏れの発見に使えます」 |
| 2:45 | ROI ダッシュボード（マイページ→利用実績） | 「比較実行も実績として計上され、対応画面数・検出指摘数から推定削減工数に反映されます」 |

## トラブル時の切り替え

- ライブクロールが遅い／失敗 → `docs/demo/sample_output/report.html` を開いて成果物ベースで説明
- ブラウザが自動で開かない → 手動で http://127.0.0.1:8765/ を開く
- ポート衝突 → `DEMO_SITE_PORT=8890 make demo`
